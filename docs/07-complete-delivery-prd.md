# 数字化交付平台完整 PRD 与三期路线

## 1. 文档定位

本文档是数字化交付平台后续开发的产品主依据，用于替代早期只围绕 `v1 MVP` 的单阶段口径。

平台正式拆为三期：

1. 一期：公司内部 BIM 资产管理试点。
2. 二期：可对外客户交付的数字化交付平台完整版。
3. 三期：增值功能、行业扩展和客户持续服务。

除非主 agent 明确更新本文档，否则开发 agent、测试 agent 和后续企业 agent 对接均以本文档为准。

## 2. 产品最终定位

数字化交付平台是一套面向 `建筑机电/BIM交付` 的单客户私有化平台，最终同时承担四类价值：

- BIM 项目资产管理：把公司或客户的大量模型、图纸、文档统一建档、检索、追踪和授权访问。
- 项目交付管理：围绕部位、节点、交付标准、资源绑定和审核流形成真实交付闭环。
- 数字档案馆：把交付成果沉淀为可查询、可预览、可审计、可移交的数字档案。
- 智慧运维底座：为后续运维、AI 检索、智能审查、构件级查询和大屏展示提供数据基础。

平台核心差异化仍是 `标准驱动交付`，但一期落地优先级调整为 `内部 BIM 资产找得到、管得住、可被企业 agent 检索`。

## 3. 阶段总览

| 阶段 | 定位 | 目标时间 | 成功标准 |
|---|---|---:|---|
| 一期 | 内部 BIM 资产管理试点 | 2-6 周 | 几百个项目可批量建档，NAS 模型可原地登记，元数据可检索，企业 agent 可读取稳定检索底座 |
| 二期 | 客户交付完整版 | 3-4 个月形成雏形，6 个月左右稳定 | 标准驱动交付、文件预览、模型轻量化、构件级能力、权限审计、客户部署和验收文档全部可用 |
| 三期 | 增值服务与持续演进 | 二期后持续 | 模型版本对比、图模联动、碰撞检查增强、AI 审核、客户侧智能问答和更多服务能力 |

## 4. 一期：内部 BIM 资产管理试点

### 4.1 一期目标

一期不追求完整数字化交付闭环，也不做真实 BIM 轻量化和构件级解析。一期目标是把公司内部几百个 BIM 项目和约 `10TB` 量级模型文件先纳入平台管理。

一期必须做到：

- 项目统一建档，可在前端增删改查。
- NAS 模型文件原地接管，不搬迁大文件。
- 数据库存储元数据，不存二进制文件。
- 文件路径、版本、专业、大小、checksum、更新时间可追踪。
- 支持项目资产台账、模型资源库、搜索筛选、容量看板。
- 下载、登记、扫描、导入、修改动作可审计。
- 企业 agent 可通过稳定读模型快速检索项目与文件资产。

### 4.2 一期功能范围

#### 项目资产台账

- 支持几百个 BIM 项目统一建档。
- 支持项目编码、项目名称、项目阶段、专业、负责人、客户/业主、建设单位、项目状态、资产来源。
- 支持前端新增、编辑、停用、查询和筛选。
- 支持后续扩展标段、楼栋、系统、子项、专业包等字段。

#### 模型资源库

- 支持按项目登记原始 BIM 模型文件。
- 支持记录 NAS 路径、平台逻辑路径、文件名、格式、大小、版本、专业、来源、登记人、更新时间、checksum、处理状态。
- 支持 `RVT`、`IFC`、`NWD/NWC`、`DWG`、`DGN`、`FBX`、`OBJ`、`GLTF/GLB`、`3D Tiles` 等格式的元数据登记。
- 一期只要求元数据可检索，不要求每种格式都能在线预览。

#### NAS 原地接管

- 新增 `StorageProvider` 抽象，一期默认实现 `NAS`。
- 支持配置 NAS 根路径、项目路径映射、扫描范围、文件扩展名白名单。
- 支持批量扫描、任务状态、失败原因、重复文件识别、重跑。
- 支持 NAS 路径变更后的重新映射能力预留。
- 一期内部可返回 NAS 路径给高权限用户和企业 agent；二期客户版默认改为平台授权访问，不暴露底层路径。

#### 批量导入

- 支持 Excel/CSV 项目清单导入。
- 支持按 NAS 目录批量生成项目、文件资源和模型资产初始记录。
- 支持导入报告，记录总数、成功数、失败数、失败原因和可重试项。

#### 版本与 checksum

- 一期制定企业内部文件版本标准。
- 默认版本来源优先级：人工指定版本 > 文件名识别 > 首次登记为 `V1`。
- checksum 用于重复文件识别、文件变更判断和企业 agent 检索增强。
- 大文件 checksum 可作为后台任务异步补齐，不阻塞首次入库。

#### 搜索与看板

- 支持按项目名、项目编码、文件名、专业、阶段、格式、版本筛选。
- 支持容量看板：项目数、模型数、总大小、专业分布、更新时间分布、Top 项目容量。
- 一期搜索深度为 `元数据 + 路径`；文档正文和模型构件属性检索放到二期或三期。

#### 企业 agent 对接底座

- 一期必须提供稳定读模型或只读宽表，不让企业 agent 直接依赖易变业务底表。
- 企业 agent 高权限读取数据库，但平台仍要保留 REST/OpenAPI 作为长期集成边界。
- 一期完成前必须再次与企业 agent 开发负责人对齐：数据库类型、字段命名、索引方式、增量同步方式、路径访问方式、是否需要正文解析。

### 4.3 一期不做

- 不做真实 BIM 在线轻量化预览。
- 不做构件级解析、构件级搜索和模型属性抽取。
- 不做完整审核流。
- 不做客户部署授权、许可和安装向导。
- 不强制完成全部交付标准闭环，但必须保留标准驱动交付底座。

## 5. 二期：客户交付完整版

二期目标是形成可交付给客户的完整私有化产品。二期不能只达到“内部能用”，必须达到客户现场可安装、可培训、可验收、可持续使用的标准。

### 5.1 标准驱动交付

- 支持项目、标段、楼栋、楼层、系统、专业、子项等完整项目结构。
- 支持公司统一交付标准库。
- 支持不同项目套用不同标准模板，并允许项目级受控调整。
- 支持部位/EBS、节点类型、交付物定义、交付物类型、交付物属性、目录模板。
- 支持标准锁定机制，锁定后不能静默破坏下游交付数据。
- 支持文件命名、格式、版本、必填属性和目录结构校验。
- 支持交付完整率、缺失项、风险项和修复入口。
- 支持生成文档交付目录、图纸交付目录和电子档案目录。

### 5.2 文件与预览

- 支持模型、图纸、文档、图片、照片、点云、视频等资源类型预留。
- 支持 PDF、图片、Office、图纸和模型预览策略。
- 支持批量下载、打包下载、下载权限控制、下载审计。
- 支持查看权限和下载权限分离。
- 支持文件路径安全映射，客户版默认不暴露底层存储路径。

### 5.3 BIM 在线预览与构件级能力

- 二期必须支持模型轻量化在线预览。
- 允许离线转换后在线预览。
- 接受接入第三方 BIM 引擎，但必须通过平台适配层接入，不绑定具体厂商。
- 支持 Web 和移动端查看。
- 支持构件级解析和构件属性查询。
- 支持构件级搜索、定位、高亮、隐藏、隔离、剖切、测量、漫游、视点保存。
- 支持多专业模型集成。
- 碰撞检查二期需支持结果接入与展示；平台内原生碰撞计算可根据引擎能力和成本裁决。

### 5.4 流程与协同

- 支持提交、审核、驳回、整改、复审、归档。
- 审核流程支持按项目配置。
- 支持任务协同、问题单、整改闭环和消息提醒。
- 支持评论、附件、人员通知、站内信，并预留企业微信或统一消息接入。
- 支持电子签章或审批意见留痕预留。
- 文件、模型、交付物和审核动作必须具备全生命周期审计。

### 5.5 权限与组织

- 角色至少覆盖平台管理员、项目管理员、BIM 经理、交付工程师、审核人、只读查看者、外部客户。
- 权限参考竞品平台，逐步支持项目级、专业级、目录级和单文件级。
- 支持下载权限与查看权限分离。
- 二期保留租户/公司/项目三级隔离能力，默认仍按单客户私有化部署。

### 5.6 查询、看板与报表

- 支持项目名、项目编码、文件名、构件属性、图纸编号、交付物名称搜索。
- 支持容量看板、项目驾驶舱、交付进度、问题数量、审核状态、模型更新和风险提醒。
- 支持 Excel/PDF 报表导出。
- 支持管理看板和领导大屏。

### 5.7 集成与部署

- 支持企业微信、LDAP/AD、OIDC 或客户统一登录扩展。
- 开放稳定 API 给企业 agent、OA、项目管理系统、档案系统、合同系统对接。
- 支持客户私有化部署、初始化向导、授权许可、演示项目模板。
- 支持备份恢复、日志诊断、健康检查、部署文档和运维手册。
- 预留国产化数据库、服务器和操作系统适配。

## 6. 三期：增值服务与持续演进

三期不阻塞二期客户交付，作为增值功能和后续服务包推进：

- 模型版本对比：构件增删改差异、属性差异、变更报告。
- 图模联动：点击图纸定位模型，点击构件反查图纸。
- 碰撞检查增强：平台内规则配置、agent 辅助审查、问题自动生成。
- AI 审核：命名、目录、属性、完整率、文档内容、模型属性的智能检查。
- 客户侧智能问答：基于交付资料、模型属性和项目知识库回答问题。
- 多行业模板：在建筑机电稳定后，再扩展市政、机场、公路、电力等行业。
- 运维资产服务：交付成果向运维对象、设备台账、巡检和维保系统延伸。

## 7. 关键非功能要求

- 私有化部署优先，默认内网可运行。
- 单项目模型文件按 `50-100GB` 规模设计，总量按 `10TB` 元数据管理设计。
- 大文件不入数据库，只存元数据、路径、checksum 和处理状态。
- 后端接口必须稳定、命名一致、错误码清晰、OpenAPI 完整。
- 所有关键接口必须具备 `traceId` 和审计记录。
- 客户交付前 P0/P1 缺陷必须清零。
- 文档必须能指导后续开发 agent、测试 agent、企业 agent 对接人员继续推进。

## 8. 阻塞级失败标准

以下问题一旦出现，视为项目不可验收：

- 用户或企业 agent 找不到已登记文件。
- 非授权用户访问到无权限项目或文件。
- 后端接口无统一规范、无文档、难以对接。
- 交付标准未定义清楚，导致后续开发和实施各自理解。
- 企业 agent 只能读业务底表，无法稳定耦合。
- 平台迁移到 Windows 后启动、构建、数据库迁移或文档入口断裂。

## 9. Windows 迁移前要求

在迁移到 Windows 开发环境前，主 agent 必须整理：

- 项目基线说明：目录结构、技术栈、端口、账号、启动顺序。
- 数据库迁移说明：Flyway 版本、样板数据、回滚风险。
- 文档入口：PRD、架构、验收、agent 对接、开发 prompt。
- 前后端启动脚本或等价命令。
- 企业 agent 对接待确认清单。
- 当前已实现能力与未实现能力清单。
