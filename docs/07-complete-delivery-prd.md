# 卓羽智能数据中台完整 PRD 与三期路线

## 1. 文档定位

本文档是卓羽智能数据中台后续开发的产品主依据，用于替代早期只围绕 `v1 MVP` 的单阶段口径。

平台正式拆为三期：

1. 一期：公司内部 BIM 资产管理试点。
2. 二期：可对外客户交付的卓羽智能数据中台完整版。
3. 三期：增值功能、行业扩展和客户持续服务。

除非主 agent 明确更新本文档，否则开发 agent、测试 agent 和后续企业 agent 对接均以本文档为准。

## 2. 产品最终定位

卓羽智能数据中台是一套面向 `建筑机电/BIM交付` 的单客户私有化平台，最终同时承担四类价值：

- BIM 项目资产管理：把公司或客户的大量模型、图纸、文档统一建档、检索、追踪和授权访问。
- 项目交付管理：围绕部位、节点、交付标准、资源绑定和审核流形成真实交付闭环。
- 数字档案馆：把交付成果沉淀为可查询、可预览、可审计、可移交的数字档案。
- 智慧运维底座：为后续运维、AI 检索、智能审查、构件级查询和大屏展示提供数据基础。

平台核心差异化仍是 `标准驱动交付`，但一期落地优先级调整为 `内部 BIM 资产找得到、管得住、可被企业 agent 检索`。

## 3. 阶段总览

| 阶段 | 定位 | 目标时间 | 成功标准 |
|---|---|---:|---|
| 一期 | 内部 BIM 资产管理试点 | 2-6 周 | 几百个项目可批量建档，NAS 模型可原地登记，元数据可检索，企业 agent 可读取稳定检索底座 |
| 二期 | 客户交付完整版 | 3-4 个月形成雏形，6 个月左右稳定 | 标准驱动交付、文件预览、模型轻量化、构件级能力、权限审计、客户部署和验收文档全部可用 |
| 三期 | 增值服务与持续演进 | 二期后持续 | 模型版本对比、图模联动、碰撞检查增强、AI 审核、客户侧智能问答和更多服务能力 |

## 4. 一期：内部 BIM 资产管理试点

### 4.1 一期目标

一期不追求完整数字化交付闭环，也不做真实 BIM 轻量化和构件级解析。一期目标是把公司内部几百个 BIM 项目和约 `10TB` 量级模型文件先纳入平台管理。

一期必须做到：

- 项目统一建档，优先通过后端 REST API 与 Swagger/OpenAPI 完成增删改查，前端页面不作为一期阻塞项。
- NAS 模型文件原地接管，不搬迁大文件。
- 数据库存储元数据，不存二进制文件。
- 文件路径、版本、专业、大小、checksum、更新时间可追踪。
- 支持项目资产台账、模型资源库、搜索筛选、容量看板。
- 登记、扫描、导入、审核、修改、删除申请和 agent 行为可审计。
- 企业 agent 可通过稳定 SQL View、REST/OpenAPI 和事件流快速检索项目与文件资产。

### 4.1.1 一期后端优先裁决

一期当前研发策略调整为 `后端数据治理优先`。在后端数据模型、导入标准、NAS 扫描、审核队列、任务状态、审计、稳定读模型和 agent 对接能力稳定前，不继续扩大前端页面范围。

一期开发采用分批交付：

1. 数据底座与 NAS 扫描入库：项目清单、NAS 路径映射、扫描任务、待审核队列、正式资产库和 SQL View 基础。
2. 真实 NAS 项目目录发现与结构适配：在不改变主架构的前提下，按真实共享盘一级目录生成项目编码建议、路径映射建议和需人工确认目录。
3. 文件类型扩展、低价值文件治理与幂等扫描：在已完成的扫描入库底座上增强真实项目资料分类、缓存/锁文件降级和重扫去重。
4. 异步任务、checksum、统计与事件流：数据库任务表、应用内 worker、checksum 异步补齐、容量统计和事件流增量同步。
5. 企业 agent、审批与受控物理删除：API Key、agent 项目范围授权、变更申请、删除审批、隔离恢复和永久删除。

新增的第 2、3 项是 Batch 1 的真实 NAS 适配增强，不改变 `Java + Spring Boot 模块化单体`、`StorageProvider`、`data_asset_*` 或 SQL View 架构边界，也不提前实现 Batch 2/3 的任务体系、API Key 或删除审批能力。

执行顺序以主 agent 最新状态文件为准。当前批次 1 已通过复验，但用户最新要求先完善批次 1 尾巴，暂不进入异步任务、checksum、统计与事件流开发；真实 NAS 适配增强当前作为批次 1 尾巴优先收口。

### 4.2 一期功能范围

#### 项目资产台账

- 支持几百个 BIM 项目统一建档。
- 支持项目编码、项目名称、项目阶段、专业、负责人、客户/业主、建设单位、项目状态、资产来源。
- 一期优先支持后端 API 新增、编辑、停用、查询和筛选；前端页面后置。
- 支持后续扩展标段、楼栋、系统、子项、专业包等字段。
- 支持 API 维护与 CSV/xlsx 批量导入两种方式初始化项目清单。

#### 模型资源库

- 支持按项目登记原始 BIM 模型文件。
- 支持记录 NAS 路径、平台逻辑路径、文件名、格式、大小、版本、专业、来源、登记人、更新时间、checksum、处理状态。
- 一期扫描白名单先固定为 `RVT`、`DWG`、`IFC`、`NWD/NWC`、`DXF`、`PDF`。
- `RVT`、`IFC`、`NWD/NWC` 默认归为 `MODEL`；`DWG`、`DXF`、`PDF` 默认归为 `DRAWING`。
- `DWG` 默认作为二维图纸治理，但人工审核时允许改为 `MODEL`。
- 一期只要求元数据可检索，不要求每种格式都能在线预览。
- 文件资产查询必须支持分页，避免几万到几十万条元数据一次性返回导致接口不可用。
- 项目、文件和容量统计查询应支持按资产来源过滤，例如只查询真实 NAS 接管来源 `NAS_REAL*`，避免历史测试数据污染一期真实试点口径。
- 真实 NAS 适配增强阶段可扩展常见项目资料类型：`DOC/DOCX` 归为 `DOCUMENT`，`XLS/XLSX` 归为 `SPREADSHEET`，`PPT/PPTX` 归为 `PRESENTATION`，`GLB/GLTF` 归为 `MODEL_VIEWER`，`ZIP/RAR` 归为 `ARCHIVE`，`WPS` 归为 `DOCUMENT`。这些类型只扩展资产治理分类，不改变模型轻量化或构件解析边界。

#### NAS 原地接管

- 新增 `StorageProvider` 抽象，一期默认实现 `NAS`。
- 支持配置 NAS 根路径、项目路径映射、扫描范围、文件扩展名白名单。
- 支持项目清单与 NAS 路径映射模板，路径映射作为项目归属的首要依据。
- 支持批量扫描、任务状态、失败原因、重复文件识别、重跑和行级/文件级报告。
- 只有命中已维护 `NAS 路径 -> 项目` 映射，或路径/文件名精确包含项目编码的扫描结果，才可高置信自动入库。
- 低置信扫描结果必须进入待审核区，人工确认项目归属、文件类型、专业和版本后再正式入库。
- 支持 NAS 路径变更后的重新映射能力预留。
- 一期内部可返回 NAS 路径给高权限用户和企业 agent；二期客户版默认改为平台授权访问，不暴露底层路径。
- 当前公司真实 NAS 根路径为 `/Volumes/zyzn/卓羽智能项目`。用户口头描述中的“卓羽智能项目文件夹”在共享盘中以该目录为准。
- 真实 NAS 首批接入采用“影子导入”：平台只读扫描 NAS，不移动、不改名、不删除原文件；先生成项目台账和路径映射建议，再按扫描置信度写入候选或正式资产。
- 真实 NAS 项目识别优先按一级目录处理，二级目录用于识别资料类型。已观察到多个项目复用 `00_工作进度`、`01_文件收发`、`02_项目资源`、`03_过程文件`、`04_共享文件`、`05_发布文件`、`06_归档文件`、`07_浏览动画` 等目录结构。
- 目录名符合 `数字-项目名` 时，项目编码优先使用数字部分，例如 `101-C塔` 使用 `101`。若数字前缀重复，不能只用数字作为唯一编码，应生成稳定短哈希编码或进入人工确认。
- 无数字前缀、弱编号、投标、参考、样板、标准、临时或专题资料目录，不得直接自动作为正式项目入库，应进入 `NEEDS_CODE_REVIEW`、`REFERENCE` 或待审核清单。
- `成果文件`、`发布文件`、`归档文件` 目录中的文件可提高归属置信度；`新建文件夹`、`临时文件`、`转换` 等目录中的文件默认降级为待审核，不自动入正式交付资产。
- `.DS_Store`、`Thumbs.db`、`desktop.ini`、`~$` 开头的 Office 锁文件、空文件和明显临时目录下的重复中间文件必须进入忽略或候选清单，不得自动入正式资产。
- 已确认的标准项目可进入正式资产库；`98`、`95`、`99` 等非标准目录、重复编号目录和疑似项目集合目录暂不导入数据库，不进入正式资产库。
- 非标准 NAS 资料治理是平台能力之一：企业 agent 接入后，平台应支持由 agent 辅助识别目录结构、项目拆分、重复编号、文件类型和导入范围，再由人工确认后导入正式资产库。
- 非标准目录在人工确认前不得进入 `ProjectAssetView`、`FileAssetView`、`ModelAssetView`，避免污染正式资产和企业 agent 检索结果。

#### 批量导入

- 支持 CSV/xlsx 项目清单导入。
- 支持 CSV/xlsx 项目清单 + NAS 路径映射导入。
- 支持通过后端 API 逐条维护项目与路径映射。
- 支持按 NAS 目录扫描生成候选文件资源，再按置信度自动入库或进入待审核。
- 支持导入报告，记录总数、成功数、失败数、失败原因和可重试项。
- 支持 NAS 一级目录发现 dry-run：输出项目编码建议、项目名称建议、路径映射建议、需人工确认目录、参考资料目录和忽略目录，不直接写入正式资产。
- 执行真实 NAS 导入时，项目台账仍落 `core_projects`，路径映射仍落 `data_asset_project_path_mappings`，扫描任务/候选/正式文件仍复用 `data_asset_scan_tasks`、`data_asset_scan_candidates`、`data_file_resources`，不新增另一套导入库。
- 重扫必须幂等：同项目同 `storage_uri` 不重复插入正式资产，只更新文件大小、mtime、处理状态、`last_verified_at` 等元数据。

#### 版本与 checksum

- 一期制定企业内部文件版本标准。
- 默认版本来源优先级：审核或人工指定版本 > 文件名保守识别 > 首次登记为 `V1`。
- 文件名版本识别支持 `V1/V2`、`RevA/RevB`、日期和常见版本号样式；无法明确识别时不得猜测，默认 `V1`。
- checksum 用于重复文件识别、文件变更判断和企业 agent 检索增强。
- 大文件 checksum 可作为后台任务异步补齐，不阻塞首次入库。
- checksum 任务采用数据库任务表 + 应用内 worker，必须可追踪状态、进度、失败原因和重试次数。

#### 专业与审核

- 专业字段采用系统内置枚举 + 项目级扩展。
- 内置专业至少包含建筑、结构、给排水、暖通、电气、消防、智能化、综合、其他。
- 待审核队列必须支持修改项目归属、文件类型、专业和版本。
- 文件真实元数据如 NAS 路径、大小、mtime、checksum 由扫描和后台任务维护，人工不得随意覆盖。

#### 搜索与看板

- 支持按项目名、项目编码、文件名、专业、阶段、格式、版本筛选。
- 支持容量看板：项目数、模型数、总大小、专业分布、更新时间分布、Top 项目容量。
- 一期搜索深度为 `元数据 + 路径`；文档正文和模型构件属性检索放到二期或三期。
- 增量同步不只依赖 `updated_at`，必须提供基于审计/事件流的变化查询能力。

#### 企业 agent 对接底座

- 一期必须优先提供 MySQL SQL View：`ProjectAssetView`、`FileAssetView`、`ModelAssetView`、`AuditEventView`。
- 一期 DB-2 首轮联调采用本机同环境：平台数据库与企业 agent / 本地 Hermes 运行在同一台机器，当前不强制依赖 shared-dev / staging。
- 企业 agent 不得直接依赖易变业务底表。
- 企业 agent 使用 API Key 接入 REST/OpenAPI，API Key 按项目范围授权，可授权全部项目或指定项目。
- agent 可查询、触发 NAS 扫描/checksum 任务、提交标注或删除申请；正式数据变更必须人工审核。
- 平台保留 REST/OpenAPI 作为长期集成边界。
- 企业 agent 接入前必须再次与开发负责人对齐：本机同环境运行方式、只读账号交付、字段命名、索引方式、事件流增量同步方式、路径访问方式、是否需要正文解析；shared-dev / staging 仅在多人协作、持续测试、演示或客户交付前验证时触发。

#### 删除与隔离

- 平台逻辑删除只删除或停用平台记录，不碰 NAS 原文件，必须审计。
- NAS 原文件物理删除必须先提交删除申请，经人工审核通过后由后端受控执行。
- 物理删除不得直接永久删除文件，必须先移动到隔离区。
- 隔离区保留期为 `30 天`，期间支持恢复；超过隔离期后才允许受控永久删除。
- 删除申请、审核、隔离、恢复和永久删除必须记录申请人、审核人、执行人、原路径、隔离路径、原因、时间、结果和失败原因。

### 4.3 一期不做

- 不做真实 BIM 在线轻量化预览。
- 不做构件级解析、构件级搜索和模型属性抽取。
- 不做完整业务交付审核流；一期只做数据治理所需的待审核、标注申请和删除审批。
- 前端资产页面不作为一期后端收口的阻塞范围；当前已补齐资产总览和项目资产详情，人工审核、导入和删除运维仍先通过 REST API + Swagger/OpenAPI 完成。
- 不做客户部署授权、许可和安装向导。
- 不强制完成全部交付标准闭环，但必须保留标准驱动交付底座。

## 5. 二期：客户交付完整版

二期目标是形成可交付给客户的完整私有化产品。二期不能只达到“内部能用”，必须达到客户现场可安装、可培训、可验收、可持续使用的标准。

### 5.1 标准驱动交付

- 支持项目、标段、楼栋、楼层、系统、专业、子项等完整项目结构。
- 支持公司统一交付标准库。
- 支持不同项目套用不同标准模板，并允许项目级受控调整。
- 支持部位/EBS、节点类型、交付物定义、交付物类型、交付物属性、目录模板。
- 支持标准锁定机制，锁定后不能静默破坏下游交付数据。
- 支持文件命名、格式、版本、必填属性和目录结构校验。
- 支持交付完整率、缺失项、风险项和修复入口。
- 支持生成文档交付目录、图纸交付目录和电子档案目录。

### 5.2 文件与预览

- 支持模型、图纸、文档、图片、照片、点云、视频等资源类型预留。
- 支持 PDF、图片、Office、图纸和模型预览策略。
- 支持批量下载、打包下载、下载权限控制、下载审计。
- 支持查看权限和下载权限分离。
- 支持文件路径安全映射，客户版默认不暴露底层存储路径。

### 5.3 BIM 在线预览与构件级能力

- 二期必须支持模型轻量化在线预览。
- 允许离线转换后在线预览。
- 接受接入第三方 BIM 引擎，但必须通过平台适配层接入，不绑定具体厂商。
- 当前尚未确定 BIM 引擎厂商；真实轻量化实施前必须先完成 `8B-0：BIM 引擎选型与接入前置评估`，输出候选引擎、格式支持矩阵、部署方式、授权成本、产物存储、Viewer 能力和 PoC 标准。
- `8A` 已收口但仅代表 Mock 适配层和安全入口，不代表真实 3D 预览已经可用。
- 支持 Web 和移动端查看。
- 支持构件级解析和构件属性查询。
- 支持构件级搜索、定位、高亮、隐藏、隔离、剖切、测量、漫游、视点保存。
- 支持多专业模型集成。
- 碰撞检查二期需支持结果接入与展示；平台内原生碰撞计算可根据引擎能力和成本裁决。

### 5.4 流程与协同

- 支持提交、审核、驳回、整改、复审、归档。
- 审核流程支持按项目配置。
- 支持任务协同、问题单、整改闭环和消息提醒。
- 支持评论、附件、人员通知、站内信，并预留企业微信或统一消息接入。
- 支持电子签章或审批意见留痕预留。
- 文件、模型、交付物和审核动作必须具备全生命周期审计。

### 5.5 权限与组织

- 角色至少覆盖平台管理员、项目管理员、BIM 经理、交付工程师、审核人、只读查看者、外部客户。
- 权限参考竞品平台，逐步支持项目级、专业级、目录级和单文件级。
- 支持下载权限与查看权限分离。
- 二期保留租户/公司/项目三级隔离能力，默认仍按单客户私有化部署。

### 5.6 查询、看板与报表

- 支持项目名、项目编码、文件名、构件属性、图纸编号、交付物名称搜索。
- 支持容量看板、项目驾驶舱、交付进度、问题数量、审核状态、模型更新和风险提醒。
- 支持 Excel/PDF 报表导出。
- 支持管理看板和领导大屏。

### 5.7 集成与部署

- 支持企业微信、LDAP/AD、OIDC 或客户统一登录扩展。
- 开放稳定 API 给企业 agent、OA、项目管理系统、档案系统、合同系统对接。
- 支持客户私有化部署、初始化向导、授权许可、演示项目模板。
- 支持备份恢复、日志诊断、健康检查、部署文档和运维手册。
- 预留国产化数据库、服务器和操作系统适配。
- `9A 客户交付准备` 必须后置到平台核心闭环、数据管家客户版、权限审计、交付包/档案目录和 BIM 引擎路线完成阶段验收之后；当前不启动客户交付包开发。

## 6. 三期：增值服务与持续演进

三期不阻塞二期客户交付，作为增值功能和后续服务包推进：

- 模型版本对比：构件增删改差异、属性差异、变更报告。
- 图模联动：点击图纸定位模型，点击构件反查图纸。
- 碰撞检查增强：平台内规则配置、agent 辅助审查、问题自动生成。
- AI 审核：命名、目录、属性、完整率、文档内容、模型属性的智能检查。
- 客户侧智能问答：基于交付资料、模型属性和项目知识库回答问题。
- 多行业模板：在建筑机电稳定后，再扩展市政、机场、公路、电力等行业。
- 运维资产服务：交付成果向运维对象、设备台账、巡检和维保系统延伸。

## 7. 关键非功能要求

- 私有化部署优先，默认内网可运行。
- 单项目模型文件按 `50-100GB` 规模设计，总量按 `10TB` 元数据管理设计。
- 大文件不入数据库，只存元数据、路径、checksum 和处理状态。
- 后端接口必须稳定、命名一致、错误码清晰、OpenAPI 完整。
- 所有关键接口必须具备 `traceId` 和审计记录。
- 客户交付前 P0/P1 缺陷必须清零。
- 客户交付准备前必须至少有一个真实项目跑通：资产接入 -> 工程主数据 -> 交付标准 -> 文件挂接 -> 审核整改 -> 交付包预检查/清单。
- 文档必须能指导后续开发 agent、测试 agent、企业 agent 对接人员继续推进。

## 8. 阻塞级失败标准

以下问题一旦出现，视为项目不可验收：

- 用户或企业 agent 找不到已登记文件。
- 非授权用户访问到无权限项目或文件。
- 后端接口无统一规范、无文档、难以对接。
- 交付标准未定义清楚，导致后续开发和实施各自理解。
- 企业 agent 只能读业务底表，无法稳定耦合。
- 平台迁移到 Windows 后启动、构建、数据库迁移或文档入口断裂。

## 9. Windows 迁移前要求

在迁移到 Windows 开发环境前，主 agent 必须整理：

- 项目基线说明：目录结构、技术栈、端口、账号、启动顺序。
- 数据库迁移说明：Flyway 版本、样板数据、回滚风险。
- 文档入口：PRD、架构、验收、agent 对接、开发 prompt。
- 前后端启动脚本或等价命令。
- 企业 agent 对接待确认清单。
- 当前已实现能力与未实现能力清单。
