# 卓羽智能数据中台竞品拆解报告

## 1. 目的与范围

本报告基于对 `https://bim.realbim.cn:13004` 已登录平台的只读审查结果整理，目标是：

- 固化现有平台的真实主链路和模块边界。
- 提炼我们必须复刻的能力底盘。
- 识别现有平台在交付标准、体验和实施路径上的可优化空间。

本次审查采用了实际登录态和真实样本项目，不是只看登录页或营销材料。

## 2. 审查样本

### 2.1 企业与项目样本

- 当前企业：`品通机电`
- 观察到的项目列表覆盖：建筑工程、公路工程、民航机场工程、水利水电工程、机电工程、市政公用工程、通信与广电工程、能源化工等
- 重点样本项目：
  - `徐州 / 数字交付`
  - `海工测试 / 二三维联动+管理对象`

### 2.2 样本项目中真实存在的资源

`海工测试` 项目中至少存在以下资源，证明平台不是空壳：

- 模型文件：`CEPB-X-2503.ubim`
- 图纸文件：`6F自控系统施工平面图.dwg`
- 其他文档：`2.4.工程数字化交付运维平台-蓝图设计汇报材料-V1.4.pptx`
- 集成对象：`压缩机`
- 管理对象：`压缩机管理对象`

## 3. 核心结论

### 3.1 平台本质

该平台本质上不是单纯的 BIM 浏览器，也不是普通网盘，而是一套把 `项目策划 + 交付标准 + 文件资源 + 模型集成 + 管理对象 + 业务协同 + 看板展示` 串联起来的工程数字化平台。

### 3.2 真实主链路

在样本项目中，平台的实际主链路可归纳为：

`项目/部位策划 -> 节点类型锁定 -> 交付物定义/类型/属性 -> 文件/模型上传 -> 模型集成 -> 管理对象 -> 文档/图纸交付视图 -> 看板/三维展示`

这个链路不是推测，是由页面状态和接口调用共同印证出来的。

### 3.3 关键产品思想

- 所有下游能力都依赖前置标准配置。
- 文件和模型不直接成为业务对象，要先变成 `资源`、`集成`、`管理对象`。
- 交付视图不是简单文件列表，而是 `与工程部位/交付标准绑定后的业务视图`。
- 3D 只是展示层，不是平台的业务核心；主数据和交付标准才是底座。

## 4. 信息架构拆解

### 4.1 顶层五大模块

平台顶部主导航包含五个一级模块：

1. `个人主页`
2. `数据管家`
3. `工程主数据`
4. `工作中心`
5. `管理中心`

这五个模块基本对应我们后续要复刻的五个产品面。

### 4.2 个人主页

可见能力：

- 待办任务
- 我的消息
- 我的分享
- 回收站
- 项目级数据概览

样本项目首页中展示了这些指标：

- 3D 模型数/总大小
- 地形数/总大小
- 图纸文档数/总大小
- 其他文档数/总大小
- 集成数
- 我的分享

判断：

- 这是项目入口和个人工作台。
- 它承担了轻量驾驶舱和代办提醒作用，但业务深度不在这里。

### 4.3 数据管家

样本项目中可见菜单：

- 文件管理
- 模型集成
- 事项列表
- 任务列表
- 导出列表
- 管理对象
- 文件服务

#### 文件管理

观察到的能力：

- 上传文件
- 新建文件夹
- 新建地形
- 导入文件目录
- 引入目录模板
- 高级搜索
- 文件状态、版本、处理耗时、创建人、创建时间

判断：

- 支持多类型文件统一收口。
- 后台存在异步处理链路，不只是原样存储。

#### 模型集成

观察到的能力：

- 新建集成
- 导入集成
- 查看集成状态、文件数量、总大小、耗时

判断：

- 平台把原始模型文件进一步加工为 “集成模型”。
- 后续三维展示很可能依赖集成发布结果，而不是直接读取原文件。

#### 管理对象

观察到的能力：

- 新建
- 修改
- 删除
- 启用

样本记录：

- `压缩机管理对象`
- 资源类型可挂接 `模型集成` 或 `资源文件`

判断：

- 管理对象是比文件更稳定的业务抽象层。
- 它很可能是后续部位、设备、对象、文档、事项联动的关键锚点。

#### 文件服务

观察到的能力：

- 新建服务
- 资源路径
- 资源类型
- 状态

判断：

- 文件服务是对外提供资源访问或业务消费能力的出口层。
- 说明平台内部有一层资源编排，而不是让业务直接查对象存储。

### 4.4 工程主数据

样本项目中可见菜单：

- 项目策划
  - 标段管理
- 交付物管理
  - 节点类型设置
  - 交付物定义
  - 交付物类型设置
  - 交付物属性设置
- 交付物关联
  - 关联模型

#### 关键观察

在多个页面中都出现相同提示：

`节点类型未锁定，请先锁定节点类型`

这说明：

- 节点类型锁定是一个强前置条件。
- 交付物定义、类型设置、属性设置、关联模型都受它控制。

#### 关联模型

样本页面中直接出现：

- 项目名 `海工测试`
- 模型名 `CEPB-X-2503.ubim`
- 操作 `自动关联`

判断：

- 工程主数据不是纸面配置，而是实际把交付物结构与模型资源绑定。
- 这正是我们后续要重点强化的 “标准驱动交付” 能力。

### 4.5 工作中心

样本项目中可以看到非常重的业务面，至少包括：

- 交付管理
  - 信息交付
  - 文档交付
    - 文档交付管理
    - 文档交付视图
  - 图纸交付
    - 图纸交付管理
    - 图纸交付视图
- 现场管理
  - 项目圈
  - 工程问题
  - 工作联系单
- 进度管理
  - 现场计划
  - 进度推演
- 质量管理
  - 工程划分
  - 质量验收
  - 质量检查
  - 试验检测
  - 质量资料总览
- 合同管理
- 构件着色

判断：

- 现有平台已经从交付底座延伸到了项目执行层。
- 但对我们 v1 来说，不需要一次性复刻全部业务深水区，先锁住交付主线即可。

### 4.6 管理中心

样本项目中可见菜单：

- 智慧大屏
- 电子沙盘
- 文件看板
- 三维推流

#### 技术观察

- `电子沙盘` 页面通过 `iframe` 挂载 `https://bim.realbim.cn:13004/file/ue/index.html`
- `文件看板` 页面存在多个 canvas 图表

判断：

- 3D 引擎是外接的展示子系统，不是业务页面直接内嵌渲染。
- 这支持我们采用 `可插拔适配层` 的架构判断。

## 5. 技术边界线索

### 5.1 前端迹象

从实际页面资源可见：

- `main/js/chunk-vendors.*.js`
- `main/js/app.*.js`
- `main/css/chunk-vendors.*.css`
- `main/css/app.*.css`
- 大量 `el-menu-item`、`el-submenu__title`、`element-icons` 样式类名

判断：

- 前端是典型的 `Vue 单页应用 + Element UI 风格`。
- 我们后续不需要技术上完全照搬，但信息架构和交互习惯可以继承。

### 5.2 后端域边界

实际接口调用中观察到以下后端域前缀：

- `busi-core-sys`
- `busi-core-msg`
- `busi-dm`
- `busi-pmd`
- `busi-pm-work`

对应推断：

- `core-sys`：平台、组织、用户、配置
- `core-msg`：消息通知
- `dm`：数据管家或数据管理
- `pmd`：工程主数据
- `pm-work`：工作中心

这和平台的页面模块边界高度一致。

### 5.3 资源与标准接口痕迹

实际观察到的代表性接口包括：

- `/api/busi-pmd/console/pmd/section/list`
- `/api/busi-pmd/console/deliverable/node/type/list`
- `/api/busi-pmd/console/deliverable/define/listByLevel`
- `/api/busi-pmd/sys/standard/config`
- `/api/busi-dm/console/model/view/release/file/list/...`
- `/api/busi-dm/console/model/file/model-file-overView`
- `/api/busi-pm-work/console/deliver/doc/file/page`
- `/api/busi-pm-work/console/deliver/drawing/file/page`
- `/api/busi-core-msg/biz/inmail/count`

判断：

- 平台已经按域拆服务或至少按域拆接口。
- 我们后续即使采用模块化单体，也应保留同样清晰的域边界和 API 命名风格。

## 6. 竞品的强项

### 6.1 强在标准前置

平台并不是让用户一上来随便传文件，而是强制从：

- 标段
- 工程部位
- 节点类型
- 交付物定义
- 属性标准
- 关联模型

这些基础配置开始。

这点非常重要，也是它能承载后续交付视图和看板的根本原因。

### 6.2 强在资源抽象

它没有把 “文件” 当作最终业务对象，而是分成：

- 文件资源
- 模型集成
- 管理对象
- 文件服务

这样后续业务才能绑定稳定对象，而不是直接绑物理文件。

### 6.3 强在多层视图

同一份底层数据可被用于：

- 文件管理页
- 文档交付管理页
- 文档交付视图页
- 文件看板
- 三维展示

说明平台在“底层资源”和“上层视图”之间做了清晰分层。

## 7. 竞品的短板与机会点

### 7.1 标准门槛高，但缺乏更好的引导

多处页面在前置条件不满足时，只给出：

`节点类型未锁定，请先锁定节点类型`

问题：

- 业务依赖是对的，但引导偏弱。
- 缺少 “下一步去哪里配置、由谁配置、还差什么” 的可操作提示。

机会：

- 我们可以把 `标准锁定状态`、`前置条件缺失清单`、`一键跳转修复` 做成平台特色。

### 7.2 交付视图依赖主数据，但可见性弱

`文档交付视图` 中出现 `未选择工程管理部位` 的状态，说明视图高度依赖部位上下文。

问题：

- 用户容易觉得页面“没内容”，但本质是配置关系没打通。

机会：

- 我们应该把 `配置依赖图` 显式展示出来，让用户知道是哪个环节没完成。

### 7.3 3D 展示与业务配置分离较远

UE 页面通过 iframe 独立挂载，工程业务系统与三维页之间的联动边界比较硬。

机会：

- 我们可以在适配层中把 `项目/部位/对象上下文` 和 `三维定位/高亮/图模联动` 做成稳定接口，不让上层业务直接依赖具体引擎页面细节。

### 7.4 模块很多，但首屏主价值不够聚焦

工作中心已扩展到进度、质量、合同等多个业务域，但对 “卓羽智能数据中台” 这一核心主题，焦点容易被稀释。

机会：

- 我们的 v1 应该先坚持 `标准驱动交付`，不在一期把所有工程管理业务都做深。

## 8. 对我们产品的直接启发

### 8.1 必须复刻的能力底盘

1. 项目和部位树管理
2. 节点类型配置与锁定
3. 交付物定义、类型、属性标准
4. 文件资源管理
5. 模型集成发布
6. 管理对象维护
7. 文档交付和图纸交付双链路
8. 看板与 3D 工作台入口
9. 审计、消息、角色权限

### 8.2 必须做得更好的地方

1. 前置条件可视化
2. 标准配置向导化
3. 交付绑定状态透明化
4. 模型、文件、管理对象三者关系可追踪
5. 私有化环境下的审计和权限收口

### 8.3 我们的 v1 差异化方向

围绕 `标准驱动交付`，优先强化以下能力：

- 部位/EBS 结构治理
- 节点类型与交付物标准锁定机制
- 交付物属性校验和缺失诊断
- 文档/图纸绑定状态与未完成项提示
- 图模联动的标准化适配接口

## 9. 结论

现有平台已经证明了一件事：卓羽智能数据中台的核心不在 “看模型”，而在 “把标准、资源、对象、视图和协同串起来”。

因此，我们的 v1 不应该从炫技 3D 或泛平台中台开始，而应该坚定从以下底盘入手：

`主数据标准化 -> 文件与模型资产化 -> 管理对象稳定化 -> 交付视图业务化 -> 3D 能力可插拔化`

这也是本次方案文档和后续开发拆分的基准线。
