# 工务署标准对齐矩阵 V0.1-beta

本矩阵用于把公司已开发的数字化交付标准、平台字段和 Hermes 回答边界回拉到深圳市建筑工务署数字化交付系列标准主干下。它不是新的标准体系，也不废弃已开发内容；它用于映射、修订和固化。

## 1. 标准来源

| source_id | 标准名称 | 版本 / 发布 | 在公司标准中的定位 |
|---|---|---|---|
| `GWS-DG-2024` | 《深圳市建筑工务署建筑工程数字化交付工作导则》 | 2024 年修订版，2024-12-31 发布，2025-01-01 实施 | 主干导则：交付节点、交付内容、交付格式、命名、交付系统和交付管理。 |
| `GWS-DG-2023` | 《深圳市建筑工务署建筑工程数字化交付工作导则》 | 2023 试行版 | 历史参考；与 2024 修订版冲突时以 2024 修订版为准。 |
| `GWS-BIM-DATA-2023` | 《深圳市建筑工务署建筑工程建筑信息模型数据交付标准》 | 2023 试行版 | BIM 数据交付、模型单元、通用/专用数据、合标性/完整性/一致性审核依据。 |
| `GWS-CONSTRUCTION-CODE-2023` | 《深圳市建筑工务署建筑工程建设阶段建筑信息模型分类和编码标准》 | 2023 试行版 | 建设阶段分类编码主干；公司不得改写其表代码和分类编码。 |
| `GWS-OM-CODE-2023` | 《深圳市建筑工务署建筑工程运维阶段建筑信息模型分类和编码标准》 | 2023 试行版，标准号 `SZGWS BIM48-2023` | 运维阶段建筑、空间、设备、关系分类编码主干；与建设阶段编码建立映射。 |

## 2. 主干原则

| 原则 | 公司落地规则 | 当前阶段 |
|---|---|---|
| 工务署分类编码保持不变 | 表代码、分类对象编码、建设阶段/运维阶段映射关系不得被公司编码覆盖。 | current |
| 公司扩展不另起主干 | 公司可扩展项目实例编码、企业资产编码、成本编码、采购编码、族库编码，但必须记录所依附的工务署分类编码。 | backlog |
| 历史资料不强制立即改名 | 历史 NAS 文件只做识别、标注、提示整改；新项目逐步执行推荐命名和 Manifest。 | current |
| catalog-only 不升级为内容理解 | 工务署对模型单元、属性和审核的要求当前只作为 future/backlog 路线，不能写成 Hermes 或平台已支持。 | current |
| 平台配置优先于 Agent 记忆 | 标准字典、专业、阶段、编码映射应进入平台配置或共享文档，不进入 Hermes memory。 | current |

## 3. 工务署导则到公司标准映射

| 工务署要求 | 公司标准映射 | 平台 / Hermes 边界 | 当前阶段 |
|---|---|---|---|
| 数字化交付以工程对象为核心，实现物理建筑和数字化建筑双交付。 | 公司标准增加工程对象、模型、文档、数据三类交付对象视角。 | 当前平台只做资产目录；不证明工程对象内容完整。 | current/backlog |
| 数字化交付包括前期规划、设计、施工 3 阶段 10 节点。 | `6.2` 以工务署节点为主干，SD/DD/CD/AS/OM 仅作公司别名。 | Hermes 只能解释目录/文件名线索，不能确认真实阶段。 | current-lineage |
| 交付内容包括三维模型、文档、数据。 | 资产分类优先级调整为模型、PDF 图纸、DWG 图纸，Manifest/Office 保留为后续能力。 | 当前只做 catalog metadata，不做正文/模型内容问答。 | current |
| 交付文件夹命名由交付节点和成果类别组成。 | 公司目录结构从真实 NAS 兼容层逐步收敛到项目 -> 阶段/批次 -> 专业 -> 资产类型。 | 历史路径只作为脱敏 path evidence。 | current-lineage/backlog |
| 三维模型命名包含项目名称、单体/子项、专业/专项、楼层/日期/版本/标段。 | 推荐 RVT 命名保留项目编号、楼栋或区域、专业、阶段、RVT、模型用途/编号、版本。 | `model_数字.rvt` 仍为弱命名，不作为标准正例。 | current-lineage |
| 图纸格式为 DWG/PDF，BIM 模型格式包括 SZ-IFC/RVT/SKP/DGN/IMODEL/OBJ/FBX/3DM/STP/STEP。 | 文件类型码表保留 RVT/PDF/DWG 为主线，并扩展 GIS、点云、视频、模型交换格式。 | 当前只识别格式，不解析内容。 | current/backlog |
| 交付系统应基于文件命名自动建立关联关系。 | 后续由 Manifest、文件名关系、模型/图纸目录关系逐步支撑。 | 当前不自动证明交付完整性。 | backlog |

## 9. Agent 边界

- Hermes 可以解释“当前标准以工务署系列标准为主干”，也可以说明某字段映射到哪个标准来源。
- Hermes 不得根据文件名或目录自动判定模型单元、构件、属性、LOD/LOI 或工务署编码合规。
- 涉及工务署 BIM 数据审核要求中的合标性、完整性、一致性时，若没有 `rvt_parse_evidence`、`component_evidence`、`full_text_evidence` 或 `manual_evidence`，必须返回 Missing Evidence。
- 工务署标准文件本身不得被当作某项目文件内容证据；它只是标准依据。
