## 1.1 当前执行路线 v2

### 一期：内部 BIM 资产管理试点

目标：

- 把公司内部几百个 BIM 项目先统一建档。
- 通过 NAS 原地接管登记模型文件元数据。
- 形成项目资产台账、模型资源库、容量看板和元数据检索。
- 输出企业 agent 可读取的稳定读模型和 OpenAPI。
- 当前执行裁决为 `后端数据治理优先`，前端资产页面暂缓，不作为一期后端验收阻塞项。

开发顺序：

1. 批次 1：数据底座与 NAS 扫描入库。
2. 批次 1 尾巴：真实 NAS 项目目录发现、文件类型扩展、低价值文件治理与幂等扫描。
3. 批次 2：异步任务、checksum、容量统计与事件流。
4. 批次 3：企业 agent、审批与受控物理删除。
5. 验收脚本：真实或模拟 NAS 目录、20 个项目导入、待审核队列、SQL View、事件流和 agent API Key 行为。

批次 1.1 与 1.2 是根据真实 NAS 结构补充的 Batch 1 增强批次，只在现有 `core_projects`、`data_asset_project_path_mappings`、`data_asset_scan_tasks`、`data_asset_scan_candidates`、`data_file_resources`、`data_asset_import_jobs`、`data_asset_import_rows` 和 SQL View 上增强导入规则，不改变项目架构。用户最新要求先完善批次 1 尾巴，因此这两个增强批次当前优先于批次 2。

### 一期后端批次拆分

#### 批次 1：数据底座与 NAS 扫描入库

目标：

- 稳定数据库结构。
- 建立项目清单、NAS 路径映射、扫描任务、待审核队列和正式资产库。
- 实现高置信自动入库、低置信进入待审核。

范围：

1. 追加 Flyway 迁移，不修改已应用迁移。
2. 项目清单维护 API。
3. NAS 路径映射维护 API。
4. API 维护与 CSV/xlsx 批量导入项目清单和路径映射。
5. NAS 扫描任务表与任务状态。
6. 扫描候选记录/待审核队列表。
7. 正式资产记录扩展。
8. 扫描白名单：`.rvt`、`.dwg`、`.ifc`、`.nwd`、`.nwc`、`.dxf`、`.pdf`。
9. `.rvt/.ifc/.nwd/.nwc` 默认归为 `MODEL`，`.dwg/.dxf/.pdf` 默认归为 `DRAWING`。
10. `.dwg` 可人工审核改为 `MODEL`。
11. 项目路径映射命中，或路径/文件名精确包含项目编码，才允许自动入库。
12. 待审核时人工可修改项目归属、文件类型、专业和版本。

#### 批次 1.1：NAS 项目目录发现与真实结构适配

目标：

- 将真实 NAS 一级目录结构转换为项目台账和路径映射建议。
- 支持 dry-run，不写正式资产，供人工确认编码和目录类型。
- 保持影子导入，只读扫描，不移动、不改名、不删除 NAS 原文件。

范围：

1. 真实 NAS 根路径以 `/Volumes/zyzn/卓羽智能项目` 为基准。
2. 一级目录作为项目或资料目录候选，二级目录作为资料类型和置信度线索。
3. 目录名前缀为 `数字-项目名` 时，项目编码优先使用数字部分。
4. 数字前缀重复时生成稳定短哈希编码或进入人工确认，不允许静默覆盖。
5. 无数字前缀、弱编号、投标、参考、样板、标准、临时或专题目录进入 `NEEDS_CODE_REVIEW`、`REFERENCE` 或忽略/待审核清单。
6. 识别常见二级目录：`00_工作进度`、`01_文件收发`、`02_项目资源`、`03_过程文件`、`04_共享文件`、`05_发布文件`、`06_归档文件`、`07_浏览动画`。
7. 首批真实试点目录为 `101-C塔`、`98-深圳口岸项目`、`99-丰图既有建模项目`。

明确不做：

- 不解析模型构件。
- 不搬迁文件到 MinIO。
- 不生成轻量化模型。
- 不改 `work-center` 交付绑定逻辑。
- 不对 NAS 做任何写入、删除、移动或改名。

#### 批次 1.2：文件类型扩展、低价值文件治理与幂等扫描

目标：

- 让真实项目中的文档、表格、汇报、展示模型和归档包具备资产分类。
- 降低系统缓存、锁文件、临时目录和中间文件误入正式资产的风险。
- 确保重扫不重复插入同一正式文件。

范围：

1. 在现有白名单基础上扩展：`DOC/DOCX -> DOCUMENT`，`XLS/XLSX -> SPREADSHEET`，`PPT/PPTX -> PRESENTATION`，`GLB/GLTF -> MODEL_VIEWER`，`ZIP/RAR -> ARCHIVE`，`WPS -> DOCUMENT`。
2. `.DS_Store`、`Thumbs.db`、`desktop.ini`、`~$` Office 锁文件、空文件不得自动入正式资产。
3. `成果文件`、`发布文件`、`归档文件` 目录中的文件可提高归属置信度。
4. `新建文件夹`、`临时文件`、`转换` 目录中的文件默认降级为待审核。
5. 同项目同 `storage_uri` 重扫只更新元数据、mtime、处理状态和 `last_verified_at`，不重复插入正式资产。
6. 审计动作覆盖目录发现、映射生成、自动入库、人工审核和拒绝。

#### 批次 2：异步任务、checksum、统计与事件流

目标：

- 建立可追踪、可重试的后端任务能力。
- 补齐 checksum、容量统计和增量同步事件。

范围：

1. 数据库任务表 + 应用内 worker。
2. 任务类型至少包括 `NAS_SCAN`、`CHECKSUM_CALC`、`QUARANTINE_CLEANUP`、`PERMANENT_DELETE`。
3. 每个任务记录状态、进度、失败原因、重试次数、创建人和更新时间。
4. checksum 异步补齐，不阻塞首次扫描入库。
5. 容量统计 API：项目数、模型数、图纸数、总容量、专业分布、Top 项目容量。
6. 增量同步采用审计/事件流，不只依赖 `updated_at`。
7. agent 或外部系统可按事件 ID 或时间窗口拉取新增、修改、审核、删除、任务状态变化等事件。

#### 批次 3：企业 agent、审批与受控物理删除

目标：

- 建立 agent 接入边界。
- 完成变更申请、删除审批、隔离删除与恢复闭环。

范围：

1. 企业 agent 使用 API Key 接入。
2. API Key 按项目范围授权，可授权全部项目或指定项目。
3. agent 可读 SQL View/只读 API，可触发扫描/checksum 任务，可提交标注或删除申请。
4. agent 不能直接修改正式资产，也不能直接物理删除 NAS 文件。
5. 逻辑删除只删除或停用平台记录，不碰 NAS 文件。
6. 物理删除必须先申请，审核通过后先移动到隔离区。
7. 隔离区保留 `30 天`，期间支持恢复；超过隔离期后才允许受控永久删除。
8. 删除申请、审核、隔离、恢复和永久删除必须完整审计。

### 二期：客户交付完整版

目标：

- 完成标准驱动交付、审核流、文件预览、模型轻量化、构件级搜索、项目驾驶舱、客户部署。
- 达到客户现场可安装、可培训、可验收、可稳定使用。

开发顺序：

1. 完整主数据与标准库。
2. 文件预览与权限下载。
3. 审核流和协同。
4. BIM 引擎适配与轻量化。
5. 构件级解析、搜索和模型操作。
6. 客户部署、备份恢复、运维诊断和交付文档。

### 三期：增值服务

目标：

- 按客户服务包逐步实现模型版本对比、图模联动、碰撞检查增强、AI 审核、智能问答和行业扩展。

## 4. 风险与控制

## 4.1 标准设计过重

风险：

- 一期花太多时间在做“大而全标准中心”，迟迟不能进入资源和视图链路。

控制：

- 只保留 `部位树 + 节点类型 + 交付物 + 属性 + 模板` 五类标准核心。

## 4.2 三维引擎反客为主

风险：

- 团队过早陷入具体引擎适配细节，影响交付底座落地。

控制：

- 一期只实现适配接口和上下文传递，不在业务平台内耦合具体引擎页面。

## 4.3 资源与对象关系混乱

风险：

- 文件、集成、对象、绑定关系命名不清，后续视图和审计无法收口。

控制：

- 所有开发任务必须严格按核心实体表述，不允许私自发明平行概念。

## 4.4 agent 并行开发冲突

风险：

- 多个开发 agent 同时修改跨域文件，产生边界污染。

控制：

- 按模块分工，尽量保证写集不重叠。
- 主 agent 统一做接口命名和跨模块协调。

## 5. 开发协作规则

- 主 agent 负责需求冻结、接口裁决、架构守门、任务排序、代码审阅。
- 开发 agent 只负责指定模块内的实现，不得自行扩 scope。
- 公共 DTO、错误码、权限模型、审计事件定义由主 agent 统一维护。
- 任一开发 agent 如需跨模块修改，必须在任务 prompt 中显式授权。
- 所有功能完成后都要回写：变更文件、接口说明、未完成项、测试结果。

## 8. 完成定义

任一模块任务只有满足以下条件才算完成：

1. 接口实现与文档契约一致
2. 关键状态机可跑通
3. 有基本测试或联调证明
4. 不破坏其他模块边界
5. 能被主 agent 直接接入下一阶段联调

## 9. 下一步执行建议

当前推荐实际开发顺序已经调整为：

1. 先冻结本次完整 PRD、架构和验收文档。
2. 继续一期内部 BIM 资产管理试点：NAS、资产台账、模型资源库、agent 读模型。
3. 一期落地前再次与企业 agent 开发负责人对齐数据库读模型和检索接口。
4. 一期验收通过后，再进入二期客户交付完整版。
5. 三期功能按客户增值需求单独排期。
