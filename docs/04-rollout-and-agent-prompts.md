## 1. 目标

本文件用于把前述方案转成可执行的研发推进蓝图，确保后续由主 agent 编排、开发 agent 实现时，模块边界、任务顺序、协作规则和验收标准一致。

## 1.1 当前执行路线 v2

### 一期：内部 BIM 资产管理试点

目标：

- 把公司内部几百个 BIM 项目先统一建档。
- 通过 NAS 原地接管登记模型文件元数据。
- 形成项目资产台账、模型资源库、容量看板和元数据检索。
- 输出企业 agent 可读取的稳定读模型和 OpenAPI。

开发顺序：

1. `data-steward.asset`：项目台账、模型资源库、批量导入。
2. `data-steward.storage`：`StorageProvider` 抽象与 NAS 扫描。
3. `data-steward.agentread`：`ProjectAssetView`、`FileAssetView`、`ModelAssetView`、`AuditEventView`。
4. 前端资产页面：项目资产台账、模型资源库、扫描任务、容量看板。
5. 验收脚本：真实或模拟 NAS 目录、20 个项目导入、agent 读模型查询。

### 二期：客户交付完整版

目标：

- 完成标准驱动交付、审核流、文件预览、模型轻量化、构件级搜索、项目驾驶舱、客户部署。
- 达到客户现场可安装、可培训、可验收、可稳定使用。

开发顺序：

1. 完整主数据与标准库。
2. 文件预览与权限下载。
3. 审核流和协同。
4. BIM 引擎适配与轻量化。
5. 构件级解析、搜索和模型操作。
6. 客户部署、备份恢复、运维诊断和交付文档。

### 三期：增值服务

目标：

- 按客户服务包逐步实现模型版本对比、图模联动、碰撞检查增强、AI 审核、智能问答和行业扩展。

## 3. MVP 验收场景

必须固定使用以下场景做联调与验收：

1. 新建一个样板项目
2. 建立一棵工程管理部位树
3. 配置并锁定一套节点类型
4. 建立一组交付物定义、类型和属性
5. 上传一份模型、一份图纸、一份文档
6. 完成一份模型集成
7. 建立一组管理对象
8. 创建一条文档交付绑定
9. 打开文档交付视图
10. 把项目上下文传入大屏和 3D 适配层

## 4. 风险与控制

## 4.1 标准设计过重

风险：

- 一期花太多时间在做“大而全标准中心”，迟迟不能进入资源和视图链路。

控制：

- 只保留 `部位树 + 节点类型 + 交付物 + 属性 + 模板` 五类标准核心。

## 4.2 三维引擎反客为主

风险：

- 团队过早陷入具体引擎适配细节，影响交付底座落地。

控制：

- 一期只实现适配接口和上下文传递，不在业务平台内耦合具体引擎页面。

## 4.3 资源与对象关系混乱

风险：

- 文件、集成、对象、绑定关系命名不清，后续视图和审计无法收口。

控制：

- 所有开发任务必须严格按核心实体表述，不允许私自发明平行概念。

## 4.4 agent 并行开发冲突

风险：

- 多个开发 agent 同时修改跨域文件，产生边界污染。

控制：

- 按模块分工，尽量保证写集不重叠。
- 主 agent 统一做接口命名和跨模块协调。

## 5. 开发协作规则

- 主 agent 负责需求冻结、接口裁决、架构守门、任务排序、代码审阅。
- 开发 agent 只负责指定模块内的实现，不得自行扩 scope。
- 公共 DTO、错误码、权限模型、审计事件定义由主 agent 统一维护。
- 任一开发 agent 如需跨模块修改，必须在任务 prompt 中显式授权。
- 所有功能完成后都要回写：变更文件、接口说明、未完成项、测试结果。

## 7. 开发 Agent Prompt 模板

以下 prompt 为后续主 agent 派工时的基线模板。

## 7.1 `core` 模块 prompt

```text
你负责数字化交付平台 v1 的 core 模块实现，只能修改 core 相关后后端代码与对应前端页面，不要改 master-data、data-steward、work-center、visualization-adapter 业务逻辑。

目标：
1. 实现项目、组织、用户、角色、权限、消息、审计基础能力。
2. 保证所有业务接口都能拿到项目上下文和当前操作者。
3. 输出统一错误码、分页结构和审计事件模型。

必须遵守：
- 后端技术栈：Java + Spring Boot
- 架构形态：模块化单体
- 权限模型：RBAC
- 关键动作必须记录 AuditLog

完成后必须回写：
- 修改了哪些文件
- 新增了哪些接口
- 还缺哪些前置依赖
- 测试结果
```

## 7.2 `master-data` 模块 prompt

```text
你负责数字化交付平台 v1 的 master-data 模块实现，只能修改主数据与标准相关代码。

目标：
1. 实现部位树、节点类型、交付物定义、交付物类型、交付物属性、目录模板、关联模型。
2. 实现“节点类型锁定”前置约束。
3. 为下游数据管家和工作中心提供可查询标准状态。

必须遵守：
- 任何下游依赖都不能绕过节点类型锁定
- 核心实体命名必须与系统设计文档一致
- 不要自行引入新的业务概念

完成后必须回写：
- 修改文件
- 接口清单
- 状态机实现说明
- 测试结果
```

## 7.3 `data-steward` 模块 prompt

```text
你负责数字化交付平台 v1 的 data-steward 模块实现，只能修改文件、模型、对象、服务相关代码。

目标：
1. 实现文件资源上传与处理状态。
2. 实现模型集成创建与发布。
3. 实现管理对象与文件服务。
4. 保证资源、集成、对象之间关系可追踪。

必须遵守：
- 不要把业务直接绑到物理文件
- 必须保留 FileResource、ModelIntegration、ManagedObject 三层抽象
- 文件处理流程要支持异步状态

完成后必须回写：
- 修改文件
- 接口与状态机
- 异步处理说明
- 测试结果
```

## 7.4 `work-center` 模块 prompt

```text
你负责数字化交付平台 v1 的 work-center 模块实现，只能修改项目首页、交付管理、事项任务和看板聚合相关代码。

目标：
1. 实现项目首页数据概览。
2. 实现文档交付管理/视图。
3. 实现图纸交付管理/视图。
4. 实现事项与任务基础能力。

必须遵守：
- 文档交付和图纸交付共享同一套主数据底座
- 当前置条件缺失时，页面要明确提示缺失项
- 不要把大屏逻辑写死在业务页面里

完成后必须回写：
- 修改文件
- 页面与接口
- 前置条件校验实现
- 测试结果
```

## 7.5 `visualization-adapter` 模块 prompt

```text
你负责数字化交付平台 v1 的 visualization-adapter 模块实现，只能修改三维适配相关代码。

目标：
1. 提供模型发布、构件定位、构件高亮、图模联动、上下文注入五类接口。
2. 保证业务平台不依赖具体引擎页面结构。
3. 提供可替换的适配器接口定义与默认空实现。

必须遵守：
- 不绑定单一 3D 引擎厂商
- 接口输入输出必须稳定
- 先做适配抽象，再做具体接入

完成后必须回写：
- 修改文件
- 适配器接口定义
- 示例实现或桩实现
- 测试结果
```

## 8. 完成定义

任一模块任务只有满足以下条件才算完成：

1. 接口实现与文档契约一致
2. 关键状态机可跑通
3. 有基本测试或联调证明
4. 不破坏其他模块边界
5. 能被主 agent 直接接入下一阶段联调

## 9. 下一步执行建议

当前推荐实际开发顺序已经调整为：

1. 先冻结本次完整 PRD、架构和验收文档。
2. 继续一期内部 BIM 资产管理试点：NAS、资产台账、模型资源库、agent 读模型。
3. 一期落地前再次与企业 agent 开发负责人对齐数据库读模型和检索接口。
4. 一期验收通过后，再进入二期客户交付完整版。
5. 三期功能按客户增值需求单独排期。

早期 `core -> master-data -> data-steward -> work-center -> visualization-adapter` 的顺序仍是模块依赖顺序，但不再代表当前业务阶段优先级。
