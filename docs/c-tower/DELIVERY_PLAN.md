# DELIVERY_PLAN: C塔平台交付计划与 Agent 集群执行方案

版本：v2.0
日期：2026-05-22
状态：总交付计划草案，供 Codex Monitor 和 Claude worker 集群执行

## 1. 交付定位

本计划面向完整 C塔智慧建筑数字底座、数据中台与可视化管理平台，不再以“能力验证切片”作为最终目标。

项目执行采用：

```text
用户
  -> Codex Monitor
  -> 需求矩阵与任务图
  -> Claude worker 集群
  -> 测试与审查门禁
  -> integration/c-tower
  -> 用户裁决 main / 独立仓库 / 定制线保留
```

Codex Monitor 负责规划、监控、验收、审查和合并裁决。Claude worker 负责在授权范围内完成具体开发、文档、测试或专项验证。

## 2. 总体阶段

| 阶段 | 周期建议 | 目标 | 主要产物 |
| --- | --- | --- | --- |
| P0 需求与方案冻结 | 2 到 4 周 | 把两份需求文件转成可执行矩阵、架构和计划 | 需求矩阵、总 PRD、技术方案、交付计划、任务图、预算口径 |
| P1 数字底座 MVP | 8 到 12 周 | 跑通平台骨架、数据中台基础、IoT 样例、主数据、API 网关和基础驾驶舱 | 管理端、数据目录、主数据、DQC、API 服务、国产库 PoC、时序 PoC |
| P2 核心业务应用 | 12 到 20 周 | 建设安防、交通、设施设备、能碳、应急、风险、通行、机器人核心闭环 | 业务模块、告警、工单、巡检、权限、移动端基础 |
| P3 数字孪生与可视化 | 16 到 24 周，可并行 | 建设驾驶舱、云渲染、BIM/GLTF、专题可视化和展示场景 | 大屏、中屏、云渲染、专题场景、模型数据管理 |
| P4 生产加固与验收 | 8 到 12 周 | 达成性能、安全、国产化、等保、灾备和交付物要求 | 压测报告、等保测评、部署手册、运维手册、验收报告 |

实际周期取决于业主接口开放、国产数据库选型、模型资产质量、云渲染资源、AI 算法采购和等保整改范围。

## 3. 组织分工

### 3.1 人类角色

| 角色 | 责任 |
| --- | --- |
| 业主接口人 | 确认需求、接口、数据、验收和部署环境 |
| 产品负责人 | 管理范围、优先级、需求变更和验收口径 |
| 架构负责人 | 裁决总体架构、数据库、接口、权限、安全和部署 |
| 项目经理 | 管理计划、资源、成本、风险和里程碑 |
| 交付负责人 | 管理部署、培训、文档、验收和运维交接 |

### 3.2 Agent 角色

| Agent | 建议使用模型 | 责任 |
| --- | --- | --- |
| Codex Monitor | Codex | 需求理解、任务图、写集管控、测试门禁、审查、合并建议 |
| Claude requirements worker | Claude | 需求矩阵、验收项、来源追踪 |
| Claude architecture worker | Claude | 技术方案、接口契约、部署方案 |
| Claude data worker | Claude | 数据模型、主数据、数据治理、数据库适配 |
| Claude backend worker | Claude | 后端服务、接口、权限、工单、业务逻辑 |
| Claude frontend worker | Claude | 管理台、大屏配置页、业务页面 |
| Claude visualization worker | Claude | 可视化专题、模型接入方案、云渲染对接 |
| Claude mobile worker | Claude | 小程序、企业微信、App 接入方案和代码 |
| Claude QA worker | Claude | 测试用例、自动化测试、验收报告 |
| Claude review worker | Claude | 代码审查、范围审查、安全和性能风险 |

### 3.3 Agent 分工原则

- Codex Monitor 不直接信任 worker 自报完成，必须看 diff、看报告、跑门禁。
- Claude worker 不直接和用户确认需求，需求变更回到 Codex Monitor。
- 每个 worker 只拥有自己的 `write_set`。
- 任何跨模块、跨数据库、跨权限、跨 API 契约的事项必须回到 Codex Monitor 裁决。
- 测试 worker 和审查 worker 默认只报告，不改业务代码。

## 4. P0 交付内容

P0 是正式开发前必须完成的阶段。

### 4.1 P0 任务

- 建立两份正式需求文件的 `REQUIREMENT_MATRIX.md`。
- 将需求按平台模块、优先级、验收方式和责任团队分类。
- 冻结国产数据库、时序数据库、对象存储、缓存、消息、云渲染和 AI 算法的候选路线。
- 输出全平台技术方案和部署架构。
- 输出 P1 到 P4 的实施预算口径。
- 输出正式 `TASK_GRAPH.yaml`。
- 验证 Claude worker 集群与 Codex Monitor 的本机运行流程。

### 4.2 P0 验收

- 需求矩阵覆盖 `数据中台.docx` 和 `智慧建筑平台清单及系统配置.xlsx`。
- 每条需求有来源、状态、优先级、实现阶段、验收标准和风险。
- 所有“待业主确认”的需求被单独列出。
- 技术方案明确哪些由当前仓库承载，哪些必须新增服务或专项。
- Agent 集群能完成至少 2 个文档任务和 1 个审查任务。

## 5. P1 交付内容

P1 建议作为对外证明能力的第一个建设版本，但必须与完整平台目标一致。

### 5.1 P1 范围

- 统一身份、菜单、角色、审计和租户基础。
- 数据源目录、主数据目录、API 服务目录。
- IoT 设备、点位、物模型和样例接入。
- 时序数据库 PoC：写入、查询、压缩、降采样。
- 国产关系数据库 PoC：核心业务表迁移和核心接口验证。
- DQC 数据质量看板：上报率、完整性、及时性、异常率。
- API 网关样例：设备状态、历史数据、控制指令、楼层能耗、空间占用。
- 基础驾驶舱：运营概览、点位概览、告警概览、能耗概览、质量概览。
- 需求覆盖矩阵后台。

### 5.2 P1 不应过度承诺

P1 可以证明能力，但不得宣称已经完成：

- 20 万点位生产级接入。
- 50,000 TPS 生产级吞吐。
- 99.99% 全平台可用率。
- 完整云渲染和数字孪生专题。
- 完整 AI 算法中台。
- 完整移动 App 多系统适配。
- 完整等保二级测评通过。

## 9. Agent 集群开发流程

### 9.1 标准流程

```text
Codex Monitor 读取 PRD/TECH_SPEC/DELIVERY_PLAN
  -> 生成或更新 TASK_GRAPH
  -> 选择 ready 任务
  -> 创建 worktree 和 Claude worker prompt
  -> Claude worker 执行
  -> worker 输出 artifacts
  -> Codex Monitor 检查 write_set 和 forbidden_paths
  -> 运行 gate commands
  -> Claude QA / review worker 辅助验收
  -> Codex Monitor 裁决返工或进入 integration/c-tower
```

### 9.2 worker 必交付物

每个 worker 必须输出到 `handoff/c-tower/<task_id>/`：

- `report.json`
- `test.log`
- `risk.md`
- `diff.patch`

`report.json` 必须包含：

- 任务 ID。
- 状态。
- 修改文件。
- 完成的验收项。
- 测试结果。
- 风险。
- 需要 Codex Monitor 裁决的事项。

### 9.3 合并规则

- worker 不允许直接合并 main。
- 通过门禁的任务先合入 `integration/c-tower`。
- `integration/c-tower` 全量门禁通过后，Codex Monitor 向用户提交合并建议。
- 用户决定进入 main、转独立仓库或保留 C塔定制线。

## 10. 门禁

### 10.1 文档门禁

- 需求来源必须可追溯。
- 不得把演示样例写成生产能力。
- 不得隐藏缺口。
- 不得绕过业主待确认项。

### 10.2 工程门禁

默认门禁：

- `git diff --check`
- 后端构建。
- 前端类型检查。
- 前端构建。
- write_set 检查。
- forbidden_paths 检查。

具体命令由 `TASK_GRAPH.yaml` 的 `gate_commands` 决定。

### 10.3 业务门禁

- 数据中台链路可追溯。
- 主数据规则可执行。
- API 网关访问可审计。
- 告警、工单、控制和移动端权限有闭环。
- 可视化专题具备数据来源和指标解释。

### 10.4 生产门禁

- 200,000 点位链路压测或专项报告。
- 50,000 TPS 容量验证或专项报告。
- 99.99% 可用性设计和演练证明。
- RTO 4 小时演练证明。
- 等保二级测评和整改证明。

## 11. 变更控制

需求变更分三类：

| 类型 | 示例 | 处理 |
| --- | --- | --- |
| 普通变更 | 文案、字段、页面布局 | Codex Monitor 记录后派 worker |
| 架构变更 | 数据库、API、权限、主数据、部署 | 必须由架构负责人和 Codex Monitor 裁决 |
| 合同变更 | 新增专题、性能指标、硬件、等保范围 | 必须进入商务和项目变更流程 |
