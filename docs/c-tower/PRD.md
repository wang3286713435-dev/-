## 1. 文档定位

本 PRD 面向 C塔业主侧定制化平台建设，目标不是做一个简单演示页，也不是只证明文件管理能力，而是规划一个能够满足当前两份正式需求文件的平台级产品。

本 PRD 的需求依据为：

## 2. 建设背景

C塔项目需要建设面向超高层智慧建筑运营、管理、服务和数字化展示的一体化平台。平台需要承载楼宇内大量物联网点位、机电设备、安防通行、能碳、交通、消防、应急、风险、机器人、移动端、可视化驾驶舱、数字孪生和数据中台能力。

当前公司既有平台底座偏向文件资产管理和内部数据治理，能够复用部分项目、文件、资产、权限、审计和基础门户能力，但不足以直接满足 C塔文件中的完整平台要求。本项目应按 C塔独立产品线进行规划，既复用现有底座，也要扩展为具备智慧建筑数字底座能力的专属平台。

## 3. 产品目标

### 3.1 总目标

建设 C塔智慧建筑数字化管理平台，形成“数字底座 + 数据中台 + 集成中台 + 业务应用 + 可视化驾驶舱 + 移动端 + 运维安全”的完整平台体系，支撑业主对 C塔建筑运营、能源、安防、交通、设施设备、应急、风险、机器人服务和可视化展示的统一管理。

### 3.2 核心目标

- 建立 C塔统一数据底座，支持物联网时序数据、BIM 空间数据、业务关系数据、视频图像和非结构化文件的多模态存储与治理。
- 建立统一主数据体系，覆盖人、设备、空间、组织、系统、接口、资产、事件和工单。
- 建立统一数据服务能力，通过 API 网关向智慧安防、智慧交通、智慧设施、智慧能碳、智慧应急、智慧风险、机器人和驾驶舱提供数据服务。
- 建设智慧建筑核心业务应用，满足清单中智慧管理系统、经营服务系统和数字底座的功能要求。
- 建设数字孪生与驾驶舱能力，支持 BIM、GLTF、实时数据融合、云渲染、专题可视化和多端展示。
- 满足国产化、等保二级、隐私保护、高并发、高吞吐、高可用、灾备恢复和可交付源代码要求。

- 安防事件发生后，平台联动通行记录、巡更记录、视频结构化数据和空间位置，辅助研判。

## 6. 功能需求

### 6.1 数字底座

#### 6.1.1 物联网平台

平台应提供统一物联网接入与设备管理能力。

必须支持：

- 产品管理：定义设备产品、物模型、属性、事件、服务和协议配置。
- 设备管理：管理设备台账、点位、状态、在线情况、楼层空间绑定和设备生命周期。
- 规则引擎：支持数据清洗、事件判断、告警触发、联动控制、消息转发和边缘规则。
- 协同管理：支持边缘网关接入、网关状态、边缘规则下发和断网本地自治。
- 远程运维：支持远程配置、日志查看、故障诊断、批量升级和运行监控。
- 安全管理：支持设备鉴权、访问控制、协议安全、证书或密钥管理。

协议适配要求：

- MQTT
- Modbus RTU/TCP
- BACnet/IP
- OPC UA
- DL/T645
- HTTP/HTTPS
- WebSocket

边缘网关要求：

- 部署于楼层弱电间或设备机房。
- 在网络中断时支持轻量级规则自治与本地联动。
- 对高频传感器数据进行本地清洗和压缩。
- 压缩率目标不低于 40%。

#### 6.1.2 数据中台

数据中台是平台核心能力，应按照“多模态存储、主数据管理、数据治理、数据服务”建设。

##### A. 多模态存储

平台不得把所有数据混放在单一数据库中，应按数据类型选择专用存储。

| 数据类型 | 存储要求 | 典型数据 |
| --- | --- | --- |
| 时序数据 | 使用时序数据库，支持高频写入、高压缩、降采样和快速历史查询 | 电表、水表、空调、环境、设备状态、传感器数据 |
| 关系数据 | 使用国产关系数据库或国产兼容数据库，支持事务一致性和主备/集群 | 用户、权限、工单、账单、设备资产、租户、业务配置 |
| 缓存数据 | 使用 Redis 集群或国产兼容缓存方案，支持毫秒级读写 | 设备实时状态、用户 Token、热点配置、临时会话 |
| 对象数据 | 使用对象存储或分布式文件存储，支持海量小文件、访问控制和生命周期 | BIM 模型、告警截图、文档、巡检照片、视频片段 |
| 空间模型数据 | 支持 BIM/IFC/GLTF/空间索引和模型元数据管理 | 楼层、房间、构件、设备空间关系 |
| 视频图像数据 | 支持图片、视频片段、结构化结果和算法分析结果存储 | 安防截图、识别结果、巡检影像 |

时序数据库必须满足：

- 单节点写入吞吐不低于 100,000 点/秒。
- 支持全楼 200,000 点位秒级采集。
- 支持列式存储和高压缩算法，压缩率不低于 10:1。
- 支持数据降采样：3 个月后秒级数据转分钟级，1 年后分钟级数据转小时级。
- 支持跨年趋势查询，目标响应时间小于 2 秒。

关系数据库必须满足：

- 优先采用国产数据库或国产兼容数据库。
- 支持主备、集群、高可用、备份恢复和事务一致性。
- 对财务、考勤、权限、工单等业务数据保证一致性和零错误口径。

缓存与对象存储必须满足：

- 热点状态和用户令牌读取达到毫秒级。
- BIM、图片、文档、巡检照片、告警截图支持权限控制、生命周期、缩略图、水印和 CDN 或等效加速能力。

##### B. 主数据管理

平台应建立统一主数据模型，至少包含 One ID、One Thing、One Map。

One ID：

- 为每个自然人建立唯一 User UUID。
- 打通门禁卡号、HR 员工号、OA 账号、企业微信或微信 OpenID。
- 支持一人一档，形成跨系统统一人员画像和轨迹分析基础。

One Thing：

- 为每个设备建立唯一 Device ID。
- 设备编码建议采用 `[系统]-[设备类型]-[楼层]-[序号]` 格式，例如 `BA-AHU-F35-01`。
- 打通 IoT 动态数据与资产静态台账。
- 台账应包含维保周期、供应商、型号、安装位置、状态、责任人、备件和维修历史。

One Map：

- 为每个空间建立唯一 Space ID。
- 空间编码应兼容 IFC、BOMA 或业主确认的空间编码规则。
- IoT 设备必须绑定 Space ID，并映射到 BIM 房间或模型构件。
- 告警、工单、人员、设备和事件应支持空间定位。

扩展主数据：

- 组织 Organization
- 租户 Tenant
- 楼栋 Building
- 楼层 Floor
- 区域 Zone
- 接口 API
- 系统 System
- 事件 Event
- 工单 Work Order
- 风险源 Risk Source
- 能耗计量 Meter
- 机器人 Robot

##### C. 数据治理

平台应提供数据清洗、质量监控、异常隔离、补数和治理看板。

必须支持：

- ETL/ELT 数据采集、转换、清洗和落库。
- 对模拟量噪声使用死区阈值过滤，例如 ±0.5% 波动不入库或按策略聚合。
- 对异常值进行隔离，例如温度 200℃ 且无烟感报警时判定为传感器异常，不进入正式统计。
- 对缺失数据进行补数，可采用线性插值或历史同期值。
- 建立 DQC 数据质量看板，展示上报率、完整性、及时性、异常率和补数情况。
- 当子系统超过 10 分钟无数据时自动生成 IT 或系统管理员工单。
- 支持数据血缘、采集日志、接口调用日志、质量规则和治理任务记录。

##### D. 数据服务

上层应用不得直接连接数据库，应统一通过 API 网关和数据服务访问。

原子数据服务：

- `getDeviceStatus`
- `getHistoryData`
- `controlDevice`
- `getPersonProfile`
- `getSpaceInfo`
- `getWorkOrderStatus`

聚合数据服务：

- `getFloorEnergy`
- `getSpaceOccupancy`
- `getBuildingSecuritySummary`
- `getEquipmentHealthSummary`
- `getEmergencySituation`
- `getCarbonEmissionSummary`

API 网关要求：

- 支持 OAuth 2.0 Access Token。
- 支持应用级 QPS 阈值、限流、降级和熔断。
- 支持接口审计日志，记录调用方、时间、耗时、参数摘要、结果和异常。
- 支持接口目录、版本管理、密钥管理、权限控制和调用统计。
- 支持对第三方系统、移动端、小程序、大屏和算法服务开放标准 API。

#### 6.1.3 集成中台

平台应建设数据集成、消息集成和服务集成能力。

必须支持：

- 数据集成：数据库、接口、文件、Excel、消息、IoT、视频结构化结果接入。
- 消息集成：支持 MQTT、WebSocket、HTTP 长轮询或其他消息协议。
- 服务集成：统一管理内部服务、第三方服务、接口凭证、回调和错误重试。
- 接入编排：支持按系统、协议、频率、数据结构和责任人管理集成任务。
- 接口状态监控：展示接口可用性、延迟、错误率和最近调用记录。

#### 6.1.4 人工智能中台

平台应建设 AI 算法与算力管理能力。

必须支持：

- 算力资源调度，支撑视频分析、图像识别、模型推理和 32B 大模型部署推理。
- 算法模型仓库，内置不少于 30 种超高层建筑场景算法能力。
- 支持第三方算法插件导入。
- 支持视频结构化分析、以图搜图和事件识别。
- 支持算法编排，通过低代码方式组合算法、规则和业务联动。
- 支持算法版本、运行状态、调用日志、效果评估和迭代优化。
- 对外开放标准 API，供业务系统、驾驶舱和智能体调用。

#### 6.1.5 位置服务中台

平台应提供位置服务和地图引擎。

必须支持：

- 定位设备管理。
- 室内地图、楼层地图、空间地图和 BIM 空间映射。
- LBS 服务，包括人员、车辆、设备、机器人和风险事件定位。
- 路径规划、无障碍路线、访客导航、停车导航和应急疏散路径。
- 定位精度目标小于等于 1 米。
- 路径计算目标小于等于 2 秒。

#### 6.1.6 统一身份认证平台

平台应建立统一身份、权限、门户和多端适配能力。

必须支持：

- 统一账号、统一登录、统一权限、统一待办和统一门户。
- 个性化工作台，支持千人千面。
- 低代码门户配置能力。
- PC、大屏、企业微信、小程序、App 多端适配。
- 支持子站嵌入。
- 预留与深湾云或业主指定平台的接口。
- 支持组织、租户、角色、岗位、菜单、数据权限、接口权限和审计策略。

## 8. 基础设施与部署要求

### 8.1 部署模式

### 8.2 服务器与存储

应按需求文件规划以下基础设施：

- 核心应用服务器：国产架构 CPU，3 台，单机内存不低于 512GB，支持高速 NVMe 缓存盘和 10G 网络。

- 驾驶舱和云渲染相关资源应支持 GPU、分布式存储和高带宽访问。
- 驾驶舱云服务建议容量应结合不少于 5 路并发操作、总存储不低于 100TB 和业主指定环境进行最终核算。

### 8.3 网络与安全设备

- 业务交换机、存储交换机和管理交换机分层规划。
- 10GE 网络接入能力。

- 内外网隔离、租户隔离、接口隔离和权限隔离。

## 12. 验收标准

### 12.1 功能验收

功能验收应以需求覆盖矩阵为准，对两份需求文件逐条标注：

- 已实现
- 部分实现
- 通过配置实现
- 通过第三方系统实现
- 待二期实现
- 不在本期范围
- 需业主确认

不得以“平台已完成”替代逐项验收。

### 12.2 数据中台验收

必须验证：

- 多模态存储按数据类型分库存储。
- 200,000 点位规模的接入方案、容量测算和压测计划明确。
- 时序数据写入、查询、压缩和降采样能力通过测试或 PoC。
- One ID、One Thing、One Map 主数据规则可执行。
- DQC 看板能展示上报率、完整性、及时性和异常率。
- API 网关能统一提供设备状态、历史数据、控制指令、楼层能耗和空间占用服务。
- 子系统 10 分钟无数据能自动触发工单或告警。

### 12.3 业务应用验收

必须验证：

- 安防、交通、设施设备、能碳、应急、风险、机器人、门户和移动端功能按模块验收。
- 告警响应、控制下发、工单闭环、巡检闭环、权限生效和消息推送符合指标。
- 业务模块与空间、设备、人员、事件、视频和工单数据打通。

### 12.4 可视化验收

必须验证：

- 驾驶舱、大屏、中屏和专题可视化按清单交付。
- BIM/GLTF 模型加载、楼层钻取、对象拾取、实时数据融合和事件联动可用。
- 云渲染并发、帧率、加载时间、图表响应和数据推送准确率符合指标。
- 数字孪生场景在指定终端上运行流畅。

### 12.5 性能与安全验收

必须验证：

- 最大接入点位、并发用户、TPS、告警响应、控制下发、BIM 加载、可用率和 RTO。
- 等保二级测评与整改。
- 隐私数据加密、脱敏、销毁和审计。
- 安全设备、日志审计、主机安全、WAF 和漏洞扫描能力。

## 13. 主 Agent 执行要求

后续开发应由主 Agent 基于本 PRD 生成 `TECH_SPEC.md`、`DELIVERY_PLAN.md` 和 `TASK_GRAPH.yaml`，再调度 worker agent 执行。

主 Agent 必须遵守：

- 先建立两份需求文件的覆盖矩阵，再拆分开发任务。
- 不得把“演示样例”宣称为“完整平台已交付”。
- 不得让 worker 自行扩大范围。
- 所有任务必须声明需求来源、验收标准、写入范围和禁止修改范围。
- 涉及数据库、权限、API、主数据、菜单和核心架构的变更必须由主 Agent 裁决。
- 每轮开发必须有测试报告、审查报告和合并报告。

## 14. 关键风险与待确认事项

| 风险或待确认项 | 影响 | 建议处理 |
