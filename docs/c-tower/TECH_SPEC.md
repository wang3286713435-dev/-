# TECH_SPEC: C塔智慧建筑数字底座、数据中台与可视化管理平台

版本：v2.0
日期：2026-05-22
状态：总技术方案草案，供主 Agent 生成任务图、专项方案和 worker prompt 使用

## 1. 技术定位

本技术方案服务于 `docs/c-tower/PRD.md` 中定义的完整 C塔平台需求，需求来源为：

- `/Users/vc/Downloads/数据中台.docx`
- `/Users/vc/Downloads/智慧建筑平台清单及系统配置.xlsx`

当前仓库可以作为 C塔平台的管理台、权限、审计、文件资产、基础数据目录和早期数据中台能力的起点，但不能被视为已经满足完整智慧建筑平台要求。完整平台需要扩展物联网接入、时序数据库、国产关系数据库、对象存储、缓存集群、集成中台、AI 中台、位置服务、数字孪生、云渲染、移动端和等保安全体系。

## 2. 总体架构

### 2.1 架构分层

平台采用分层架构：

| 层级 | 主要能力 |
| --- | --- |
| 访问层 | PC 管理端、大屏、中屏、B 端小程序、C 端小程序、移动 App、开放 API |
| 业务应用层 | 智慧安防、智慧交通、设施设备、能碳、应急、风险、通行、机器人、租户服务 |
| 数字底座层 | 物联网平台、数据中台、集成中台、AI 中台、位置服务中台、统一身份认证 |
| 数据存储层 | 国产关系数据库、时序数据库、Redis 或兼容缓存、对象存储、模型存储、日志存储 |
| 集成与消息层 | API 网关、消息总线、事件总线、协议适配、第三方系统连接器 |
| 基础设施层 | 国产服务器、网络、安全设备、GPU/云渲染、分布式存储、监控运维 |

### 2.2 推荐服务边界

完整平台不建议全部堆在当前模块化单体内。建议采用“当前平台底座 + C塔专属服务群”的演进路线。

| 服务或模块 | 建议形态 | 说明 |
| --- | --- | --- |
| `delivery-core` | 复用并谨慎扩展 | 用户、权限、菜单、审计、租户和操作日志 |
| `delivery-data-steward` | P1 可复用，后续可拆分 | 数据源目录、主数据、数据质量、API 服务目录 |
| `delivery-app` | 复用启动与迁移机制 | 仅承载当前 Java 服务，Flyway 迁移只能追加 |
| `ctower-iot-service` | 新增独立服务 | 设备接入、点位、物模型、规则、边缘网关 |
| `ctower-data-platform-service` | 新增独立服务 | 时序、主数据、治理、数据服务、DQC |
| `ctower-integration-gateway` | 新增独立服务 | 协议适配、数据集成、消息集成、服务集成 |
| `ctower-business-services` | 按领域拆分 | 安防、交通、设施、能碳、应急、风险、机器人 |
| `ctower-visualization-service` | 新增独立服务 | 驾驶舱、专题可视化、模型管理、云渲染调度 |
| `ctower-mobile-gateway` | 新增独立服务 | 小程序、企业微信、App、消息推送、移动安全 |
| `ctower-ai-platform-adapter` | 新增独立服务 | 算法仓库、算法编排、视频结构化、大模型 API |
| `ctower-ops-security` | 新增运维能力 | 监控、日志、审计、等保、备份、灾备 |

### 2.3 当前仓库适配边界

当前仓库适合承担：

- P0 需求矩阵、技术方案、任务图和 Agent 调度试点。
- P1 管理台、数据目录、主数据目录、接口目录、数据质量看板和基础权限。

当前仓库不应单独承担：

- 完整移动 App 多系统适配。
- 完整等保二级软硬件闭环。

## 3. 数据架构

### 3.1 多模态存储

平台必须按数据类型采用不同存储，不允许把所有数据放入单一业务库。

| 数据域 | 推荐技术路线 | 关键要求 |
| --- | --- | --- |
| 业务关系数据 | 达梦、人大金仓、OceanBase MySQL 兼容模式或业主指定国产库 | 事务一致性、主备或集群、备份恢复 |
| 时序数据 | TDengine 或业主认可的国产时序数据库 | 100,000 点/秒单节点写入、200,000 点位秒级采集、压缩和降采样 |
| 热点缓存 | Redis 集群或国产兼容缓存 | 设备实时状态、Token、热点配置，毫秒级读写 |
| 对象文件 | 对象存储或分布式文件系统 | BIM、GLTF、文档、巡检照片、告警截图、视频片段 |
| BIM 与空间模型 | 模型存储 + 元数据关系库 + 空间索引 | 楼层、房间、构件、设备和 Space ID 映射 |
| 日志与审计 | 日志平台或关系库归档 | 调用日志、操作日志、安全审计、设备接入日志 |
| AI 结构化结果 | 关系库 + 对象存储 | 视频结构化、人脸脱敏索引、图片检索结果和算法事件 |

### 3.2 主数据模型

核心主数据：

- One ID：统一人员 ID，映射门禁卡、HR、OA、企业微信、微信 OpenID。
- One Thing：统一设备 ID，编码示例 `BA-AHU-F35-01`，连接 IoT 动态点位和资产台账。
- One Map：统一空间 ID，兼容 IFC/BOMA 或业主确认规则，绑定 BIM 房间和楼层空间。

扩展主数据：

- Organization
- Tenant
- Building
- Floor
- Zone
- Device Product
- Device Point
- API Service
- Risk Source
- Emergency Plan
- Work Order
- Visualization Scene

主数据约束：

- 设备、点位、空间、人员和告警必须具备可追溯唯一 ID。
- 所有跨系统映射必须保留来源系统、来源 ID、同步时间和可信等级。
- 任何 worker 不得自行修改主数据唯一性规则，必须由 Codex Monitor 或主 Agent 裁决。

### 3.3 数据治理链路

数据治理链路：

```text
采集接入 -> 协议解析 -> 边缘清洗 -> 消息队列 -> 落库 -> 治理规则 -> DQC 看板 -> 数据服务
```

治理规则必须覆盖：

- 死区过滤：模拟量小幅波动可按 ±0.5% 阈值过滤或聚合。
- 异常隔离：例如温度 200℃ 且无烟感报警时隔离为传感器异常。
- 缺失补数：支持线性插值或历史同期补数。
- 上报监控：系统超过 10 分钟无数据时生成告警或工单。
- 质量指标：上报率、完整性、及时性、异常率、补数率。

### 3.4 API 服务架构

所有上层应用必须通过 API 网关访问数据服务，不得直接连接底层数据库。

API 服务分三类：

| 类型 | 示例 | 说明 |
| --- | --- | --- |
| 原子服务 | `getDeviceStatus`、`getHistoryData`、`controlDevice` | 直接面向设备、点位、空间、人员 |
| 聚合服务 | `getFloorEnergy`、`getSpaceOccupancy`、`getEmergencySituation` | 面向业务应用和驾驶舱 |
| 开放服务 | 移动端、第三方系统、业主平台接口 | 需要应用鉴权、限流、审计和版本控制 |

API 网关必须支持：

- OAuth 2.0 Access Token。
- 应用级 QPS 阈值。
- 限流、降级、熔断。
- 接口目录、版本、密钥、权限和调用统计。
- 审计日志：调用方、时间、耗时、参数摘要、结果和异常。

## 4. 业务应用架构

### 4.1 智慧管理系统

| 领域 | 后端能力 | 前端能力 | 关键集成 |
| --- | --- | --- | --- |
| 智慧安防 | 告警、通行、巡更、视频结构化事件 | 安防监控台、告警中心、权限管理、研判页面 | 门禁、视频、位置、BIM、One ID |
| 智慧交通 | 停车、车位、充电、预约、违停 | 交通监控台、停车地图、车流热力 | 车场、充电桩、导航、租户 |
| 设施设备 | 设备台账、工单、巡检、备件、健康评分 | 机电监控台、资产、工单、维修日历 | IoT、One Thing、One Map |
| 能碳 | 能耗、供冷、照明、环境、碳资产 | 能源监控台、策略模拟、碳排分析 | 时序库、计量表、空间、租户 |
| 应急指挥 | 预案、资源、疏散、演练、复盘 | 应急沙盘、BIM 透明楼体、回放 | 消防、视频、位置、通信 |
| 风险管理 | 风险源、监测、识别、处置闭环 | 风险清单、风险地图、处置分析 | BIM、AI 视频、工单 |

### 4.2 经营服务系统

| 领域 | 关键能力 |
| --- | --- |
| 智慧通行服务 | 访客预约、审批、二维码、门禁授权、到访离场 |
| 智慧机器人服务 | 机器人调度、路线、任务、告警、充电和服务记录 |
| 统一门户服务 | C 端小程序、租户后台、服务申请、消息、待办、导航 |

### 4.3 可视化应用

可视化服务应独立于普通管理后台，采用专题场景管理。

核心能力：

- BIM/GLTF 模型管理。
- 场景配置。
- 实时数据绑定。
- 告警和事件叠加。
- 云渲染调度。
- 大屏、中屏、Web 和移动端展示适配。

对高性能数字孪生场景，应单独建设可视化专项，不能由普通前端页面直接承诺。

## 6. 安全架构

### 6.1 身份与权限

必须支持：

- 统一身份认证。
- 角色、菜单、按钮、数据、接口和租户权限。
- PC、大屏、移动端和 API 权限一致。
- 高风险操作二次确认、审批或多因素认证。

### 6.2 隐私保护

必须支持：

- 人脸数据加密存储。
- 手机号、人脸、证件号等敏感信息脱敏展示。
- 访客、人脸和轨迹数据按策略自动销毁。
- 个人信息访问、导出、删除和授权留痕。

### 6.3 等保二级

必须支持：

- 漏洞扫描。
- 日志审计。
- 主机安全。
- WAF。
- 安全策略配置。
- 安全测评整改闭环。

## 7. 性能与可靠性架构

目标指标：

| 指标 | 目标 |
| --- | --- |
| 最大接入点位 | 不低于 200,000 点 |
| 最大并发用户 | 不低于 2,000 人 |
| 数据吞吐量 | 不低于 50,000 TPS |
| 报警响应时间 | 小于等于 1 秒 |
| 控制指令下发 | 小于等于 1 秒 |
| BIM 模型加载 | 小于等于 5 秒 |
| 年可用率 | 不低于 99.99% |
| RTO | 小于等于 4 小时 |

实现要求：

- IoT 接入、消息总线、时序库和告警链路必须单独压测。
- API 网关、业务服务和数据库必须做容量测算。
- 关键服务主备或集群部署。
- 数据库、对象存储和配置必须有备份恢复策略。
- 生产指标不得只靠开发环境测试证明。

## 9. 与 Agent 集群的工程边界

本项目后续开发采用“Claude worker 集群 + Codex Monitor”模式。

Codex Monitor 负责：

- 读取 PRD、TECH_SPEC、DELIVERY_PLAN 和 TASK_GRAPH。
- 拆任务、分配任务、限制写集、检查冲突。
- 创建或审核 worktree、分支和 worker prompt。
- 验证 Claude worker 的 diff、报告、测试和风险。
- 决定返工、挂起、进入 integration 分支或提交用户裁决。

Claude worker 负责：

- 在授权 worktree 中完成具体需求、代码、测试或文档。
- 严格遵守 `write_set` 和 `forbidden_paths`。
- 输出 `report.json`、`test.log`、`risk.md`、`diff.patch`。
- 不直接合并 main，不自行扩展需求。

## 11. 不可越界规则

- 任何文档或页面不得宣称当前平台已经完成完整 C塔智慧建筑平台。
- 任何 worker 不得绕过 API 网关直接让业务应用读取生产数据库。
- Flyway 迁移只能追加，不得修改已应用迁移。
- 主数据唯一性规则、权限模型、API 合约、数据库主约束必须由 Codex Monitor 或主 Agent 裁决。
- 可视化、AI、移动端、等保、国产数据库和时序数据库必须按专项验收，不得用普通页面替代验收。
