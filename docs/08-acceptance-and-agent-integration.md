# 验收矩阵与企业 Agent 对接要求

## 1. 文档定位

本文档固定数字化交付平台三期验收口径，并单独锁定企业内核级 agent 与平台对接的要求。

企业 agent 当前仍处于 MVP 阶段。一期平台必须先提供稳定、易读、可增量同步的数据契约，避免未来 agent 被平台业务表结构变化拖垮。

## 2. 一期验收矩阵：内部 BIM 资产管理试点

| 验收域 | 必测项 | 通过标准 | 阻塞级别 |
|---|---|---|---|
| 项目台账 | 导入不少于 20 个真实项目 | 项目编码、名称、阶段、专业、负责人、客户/业主可查 | P0 |
| 项目维护 | 前端增删改查项目 | 普通维护人员可完成建档、编辑、停用、筛选 | P0 |
| NAS 接管 | 扫描真实 NAS 目录 | 正确识别文件名、路径、格式、大小、项目归属 | P0 |
| 文件登记 | 大文件不入库 | 数据库只存元数据、路径、checksum、状态 | P0 |
| 模型资源库 | 单项目多模型登记 | 单项目 50-100GB 量级模型元数据可管理 | P0 |
| 搜索筛选 | 元数据 + 路径检索 | 可按项目名、项目编码、文件名、专业、阶段检索 | P0 |
| checksum | 重复/变更识别 | checksum 可异步补齐，重复文件可识别或标记 | P1 |
| 版本 | 企业内部版本规则 | 人工版本、文件名版本、默认 V1 规则明确 | P1 |
| 容量看板 | 10TB 量级元数据统计 | 项目数、模型数、总容量、专业分布、更新时间可查 | P1 |
| 下载 | 下载或打开入口 | 内部用户可通过平台或 agent 找到真实文件路径 | P0 |
| 权限 | 项目访问隔离 | 普通用户不能访问无权限项目资产 | P0 |
| 审计 | 关键动作留痕 | 导入、扫描、下载、修改、删除均可审计 | P0 |
| 任务 | 扫描任务可追踪 | 任务可查看状态、失败原因、重跑结果 | P0 |
| Agent | 稳定读模型可用 | agent 可读项目、文件、模型、审计视图，不依赖业务底表 | P0 |

## 3. 一期 MVP 场景

一期必须用真实或接近真实的数据完成以下闭环：

`20 个项目清单 -> NAS 根目录配置 -> NAS 扫描任务 -> 项目资产台账 -> 模型资源库 -> checksum/版本登记 -> 容量看板 -> 企业 agent 查询项目和文件 -> 审计日志可查`

额外要求：

- 后续必须支持从 20 个项目扩展到公司全量几百个项目。
- 总规模按 `10TB` 文件资产元数据设计。
- 若 NAS 路径失效，平台必须能提示失效原因，不允许静默显示可用。

## 4. 企业 Agent 对接契约

### 4.1 对接原则

- 企业 agent 权限很高，可以把数据库检索作为自身能力之一。
- 平台仍需提供稳定读模型，不能让 agent 直接耦合频繁变化的业务底表。
- 一期先满足 `元数据 + 路径` 检索。
- 文档正文检索、模型属性检索、构件级检索放到二期或三期。
- 平台保留 REST/OpenAPI，作为后续 agent 工具调用和客户交付接口。

### 4.2 稳定读模型

一期应向企业 agent 固化以下只读视图或只读宽表。实际数据库对象可用 SQL View、物化表或同步表实现，但字段含义必须稳定。

| `discipline_scope` | 专业范围 |
| `manager_name` | 项目负责人 |
| `owner_org_name` | 客户/业主 |
| `asset_status` | 资产状态 |
| `model_file_count` | 模型文件数量 |
| `total_size_bytes` | 模型总大小 |
| `last_asset_updated_at` | 最近资产更新时间 |

### 4.3 REST/OpenAPI 保留接口

一期需保留以下 API 能力，后续企业 agent 可作为工具调用：

- 项目资产查询。
- 文件资产查询。
- 模型资产查询。
- NAS 扫描任务查询。
- 容量统计查询。
- 文件访问或下载入口。
- agent 元数据搜索接口。

接口必须满足：

- 统一响应结构。
- 统一错误码。
- 响应包含 `traceId`。
- OpenAPI 文档可访问。
- 权限与审计策略明确。

### 4.4 增量同步要求

为避免 agent 每次全量扫描数据库，一期必须预留：

- 所有读模型具备 `updated_at`。
- 逻辑删除数据具备可识别状态或删除事件。
- 扫描任务、导入任务、文件变更具备审计或事件记录。
- 可按时间窗口拉取变更数据。
- checksum 变化可作为文件内容变化的重要信号。

### 4.5 一期落地前再次确认项

在一期验收前，主 agent 必须再次向企业 agent 开发负责人确认：

- 企业 agent 使用的数据库连接方式和只读账号策略。
- 是否直接读 MySQL，还是读同步库、搜索引擎或向量库。
- 是否需要平台推送变更事件。
- 是否需要 agent 读取真实 NAS 路径。
- 是否需要解析 PDF/Office 正文。
- 是否需要 agent 触发平台扫描任务。
- 是否需要 agent 调用平台下载接口。

## 5. 二期验收矩阵：客户交付完整版

| 验收域 | 必测项 | 通过标准 | 阻塞级别 |
|---|---|---|---|
| 标准驱动 | 部位、节点、交付标准完整配置 | 可套用模板、锁定标准、校验缺失项 | P0 |
| 项目结构 | 标段、楼栋、楼层、系统、专业 | 资源可绑定到正确工程结构 | P0 |
| 文件交付 | 文档/图纸/模型统一归集 | 文件可分类、预览、绑定、审核、交付 | P0 |
| 文件预览 | PDF/Office/图片/图纸预览 | 常见文件可打开，失败有明确提示 | P0 |
| BIM 预览 | 模型轻量化在线预览 | 支持离线转换后 Web 查看 | P0 |
| 构件能力 | 构件解析、搜索、定位、高亮 | 构件属性可查，搜索可定位到模型 | P0 |
| 视图工具 | 隐藏、隔离、剖切、测量、漫游、视点 | 基础查看操作流畅可用 | P1 |
| 多专业 | 多专业模型集成 | 建筑、结构、机电等可集成展示 | P1 |
| 碰撞检查 | 碰撞结果接入 | 可查看结果并关联问题或整改项 | P1 |
| 审核流 | 提交、审核、驳回、整改、归档 | 不同项目可配置流程 | P0 |
| 权限 | 查看/下载分离 | 无权限访问被拒绝且有审计 | P0 |
| 报表 | Excel/PDF 导出 | 交付进度、完整率、容量可导出 | P1 |
| 大屏 | 管理看板和领导大屏 | 展示资产、交付、问题、风险指标 | P1 |
| 移动端 | 模型和文件查看 | 平板/移动端关键查看流程可用 | P1 |
| 部署 | 私有化部署 | 安装、初始化、备份、恢复、健康检查可执行 | P0 |
| 文档 | 客户交付文档包 | 部署、运维、用户、API、验收文档齐全 | P0 |

## 6. 二期完整验收主链路

二期必须通过以下完整链路：

`客户环境部署 -> 初始化项目 -> 套用交付标准 -> 建立部位/专业/标段 -> 导入文档/图纸/模型 -> 模型轻量化 -> 构件解析 -> 文件和构件绑定 -> 提交审核 -> 整改闭环 -> 文档/图纸/模型交付视图 -> 项目驾驶舱 -> 报表导出 -> 审计追踪`

## 7. 三期验收原则

三期功能作为增值服务单独验收，不阻塞二期客户交付：

- 模型版本对比按项目或模型单独验收。
- 图模联动按图纸类型、构件映射质量和定位准确率验收。
- AI 审核按规则覆盖率、召回率、误报率和人工复核体验验收。
- 客户侧智能问答按资料召回、引用准确性、权限隔离和审计验收。

## 8. 缺陷分级与发布门槛

| 等级 | 定义 | 发布要求 |
|---|---|---|
| P0 | 数据丢失、权限泄漏、系统不可用、文件找不到、核心链路断裂 | 不允许发布 |
| P1 | 核心功能错误、预览不可用、审核流错误、接口不兼容 | 客户交付前必须清零 |
| P2 | 体验问题、边界错误、非核心统计偏差 | 可带计划发布，但必须记录 |
| P3 | 文案、样式、低频优化 | 可进入后续迭代 |

## 9. 交付文档验收

每期交付至少包含：

- PRD 与范围说明。
- 架构设计与接口契约。
- 数据库字段字典和迁移说明。
- 测试矩阵和验收报告。
- 部署和启动说明。
- 开发 agent prompt 和测试 agent prompt。
- 已知问题和后续计划。

二期客户版额外必须包含：

- 客户部署手册。
- 运维手册。
- 备份恢复手册。
- 用户操作手册。
- API 对接手册。
- 演示项目模板。
