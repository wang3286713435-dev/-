可以开始第一阶段开发，但还需要把“工程基线”显式执行掉。换句话说，业务文档已经够用了，真正还差的是一组会直接影响开发效率和返工率的开工动作。

本文件把这些动作拆成可执行 backlog，后续主 agent 可以按此直接派发。

## 2. 当前还需要完善的重点

在准备开工这件事上，当前还需要补齐四类东西：

1. 工程骨架：仓库结构、模块工程、前端壳子、本地依赖环境
2. 平台底座：认证、项目上下文、统一返回、错误码、审计、追踪
3. 研发规范：命名、分层、数据库迁移、接口文档、联调方式
4. 样板数据：至少要有一套能支撑联调的样板项目和初始化脚本

## 3. 第一阶段建议拆分

### 3.1 P0：仓库与环境初始化

目标：

- 让所有开发 agent 在同一工程骨架上工作

任务：

1. 初始化 Git 仓库
2. 创建 `backend/`、`frontend/`、`infra/`、`scripts/` 目录
3. 创建后端 Maven 父工程和五个业务模块
4. 创建前端 Vue 3 工程和模块目录
5. 创建 `docker-compose`，拉起 MySQL、Redis、MinIO
6. 创建 `.env.example`、本地启动说明和端口约定

完成标准：

- 后端可以空启动
- 前端可以空启动
- 本地依赖可以一键启动

### 3.2 P0：平台公共底座

目标：

- 建立所有后续模块都会依赖的共性能力

任务：

1. 登录、登出、刷新 token、当前用户接口
2. 项目切换接口和项目上下文拦截
3. 统一返回结构和全局异常处理
4. `traceId` 拦截器
5. `AuditLog` 事件模型和审计切面
6. 统一分页 DTO
7. 基础 RBAC 表结构

完成标准：

- 前端成功登录后可拿到当前用户和当前项目
- 任意接口返回都符合统一规范
- 关键接口能记录审计

### 3.3 P1：前端中后台骨架

目标：

- 让后续业务页面都有统一壳子，不再重复搭页面底座

任务：

1. 登录页
2. 主框架布局
3. 顶部项目切换
4. 左侧按权限动态菜单
5. 全局消息入口
6. 面包屑和标签页导航
7. 空状态、权限缺失、前置条件缺失通用组件

完成标准：

- 能进入登录页和主框架
- 菜单可按角色控制
- 页面可展示“节点类型未锁定”等前置条件提示

### 3.4 P1：样板数据与联调底座

目标：

- 提前建立一套稳定的联调对象，避免接口一做完就没数据可看

任务：

1. 初始化样板组织、用户、角色
2. 初始化一个样板项目
3. 初始化项目基础字典
4. 预置一套最小菜单权限
5. 预置一套最小审计事件定义

完成标准：

- 新环境启动后，可以直接登录并进入样板项目

## 4. 开发顺序建议

建议按下面顺序推进，不要倒着来：

1. 仓库与依赖环境
2. 后端 `core` 骨架
3. 前端中后台骨架
4. 认证、项目上下文、RBAC、审计
5. 样板数据
6. 之后再进入 `master-data`

原因很简单：如果 `core` 和前端壳子不稳，后面每个业务域都会重复返工。

## 5. 第一阶段完成定义

第一阶段不是“建好了所有业务”，而是达到下面这个程度：

1. 代码仓库结构稳定
2. 前后端工程都能本地启动
3. 本地依赖环境一键启动
4. 登录、项目切换、权限控制可用
5. 审计和追踪打通
6. 前端具备完整中后台页面壳子
7. OpenAPI 和数据库迁移机制就位
8. 样板数据可支持下阶段联调

## 6. 仍建议你拍板的少量事项

如果没有特别要求，我们就按推荐值直接推进：

### 6.1 认证模式

推荐：`本地账号密码 + JWT`  
影响：最快落地，后续可平滑接企业 SSO

### 6.2 基础设施默认实现

推荐：`MySQL 8 + Redis 7 + MinIO`  
影响：开发效率最高，私有化也容易落地

### 6.3 一期租户策略

推荐：`单租户部署，多项目隔离`  
影响：减小一期复杂度，但保留未来多租户字段

### 6.4 样板项目命名

推荐：`SAMPLE-MEP-001 / 机电交付样板项目`
