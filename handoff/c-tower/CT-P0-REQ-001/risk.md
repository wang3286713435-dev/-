## 3. 可能影响后续 P1 backlog 的风险

### 3.1 架构风险
- **多模态存储架构复杂度**：同时引入时序库、国产关系库、Redis、对象存储、模型存储和日志存储，架构协调和数据一致性保障难度高。建议 P1 优先做多模态存储 PoC。
- **当前单体架构到微服务群的拆分**：当前可能为模块化单体，需拆分为 IoT/数据/集成/业务/可视化等多个独立服务，拆分策略和顺序直接影响 P1 交付。
- **API 网关为新增组件**：当前无统一 API 网关，所有应用直接访问数据库的模式需在 P1 改造，影响面大。

### 3.2 数据风险
- **200,000 点位链路未经验证**：P0 仅做了需求映射，P1 必须完成时序库+消息链路 PoC，否则无法支撑后续业务模块开发。
- **主数据规则依赖业主数据**：One ID/One Map 的映射需要业主提供门禁、HR、OA、BIM 等系统的实际数据格式，数据质量不可控。
- **数据治理规则调优周期长**：死区阈值、异常判定、补数策略等需要积累实际运行数据才能调优。

### 3.3 资源风险
- **硬件采购周期**：国产服务器、存储、交换机、安全设备的采购和到货周期可能长达 8-12 周，P1 必须提前启动。
- **可视化专项团队**：数字孪生、云渲染和 UE5 开发需要专业技能，现有前端团队可能不具备。
- **AI 算力和算法**：32B 大模型部署推理和 30+ 算法需要 GPU 资源，采购和部署周期需提前规划。

### 3.4 范围风险
- **需求文件范围大**：198 条需求映射，若全部作为 P1 则周期不可控。建议 P1 聚焦数字底座 MVP，业务应用放 P2。
- **可视化需求"初步"性质**：源文件明确标注需求为初步，后续可能大幅扩展，需在合同中约定变更控制机制。
- **课题研究类可视化**（人机协同起重、安全设计、健康监测、碳排放）带有研究性质，产出不确定，建议作为独立课题而非平台功能。

### 3.5 合规风险
- **等保二级费用**：中标方承担全部费用，需在 P0 预算中单列测评、整改和安全组件费用。
- **个人信息保护**：人脸加密、脱敏、定期销毁等功能涉及法规合规，需法务和 DPO 审查。
- **开源协议合规**：源代码交付包含开源组件时需提供完整 SBOM 和许可证清单。
