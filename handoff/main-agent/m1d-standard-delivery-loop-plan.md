# M1D：标准驱动交付闭环强化计划

更新时间：2026-05-20

## 1. 批次定位

`M1D：标准驱动交付闭环强化` 是 M1C 之后的平台本体稳定批次。

M1D 的目标不是继续新增大模块，而是把当前已有的工程主数据、文档交付、图纸交付、批量挂接、审核、整改、导出预检查等能力串成普通员工可执行的完整闭环。

一句话目标：

> 不依赖 Hermes，员工也能从一个真实项目进入交付页面，看清应交项和缺失项，选择文件补交，提交审核，处理整改，复审后看到完整率刷新，并完成交付包 dry-run 预检查。

## 2. 当前问题

1. 标准、挂接、审核、整改和预检查能力已经存在，但用户仍可能不知道完整流程顺序。
2. 文档交付和图纸交付页面需要更明确表达：
   - 哪些是应交项。
   - 哪些是缺失项。
   - 哪些已挂接但待审。
   - 哪些被驳回需要整改。
   - 哪些能纳入交付包预检查。
3. 批量挂接、审核、整改、复审之后，完整率和状态刷新必须稳定。
4. 真实项目 105/503 不能作为唯一特例，至少再验证一个真实 NAS 项目。

## 3. 本批必须完成

### A. 交付页面流程引导

文档交付和图纸交付页面需要让业务用户看懂：

- 当前项目标准底座是否就绪。
- 当前视图应交多少、已交多少、缺失多少、待审多少、已驳回多少。
- 下一步该做什么：
  - 去补标准。
  - 去补交文件。
  - 提交审核。
  - 处理整改。
  - 做导出预检查。

### B. 缺失项与挂接闭环强化

缺失项必须能解释：

- 缺的是哪个交付物定义。
- 缺的是哪种文件类型。
- 目标是部位还是对象。
- 为什么当前算缺失。
- 可以从当前项目资产目录选择哪些文件补交。

文件选择必须保持分页/远程查询，不能回退成全量加载。

### C. 审核、整改、复审闭环

必须验证并必要时修复：

- 挂接后状态可提交审核。
- 审核通过后完整率刷新。
- 审核驳回后产生整改项。
- 整改项可处理、关闭、重新打开。
- 重新补交或复审后状态能回到交付链路。
- 审核记录和整改记录可追踪。

### D. 交付包 dry-run 预检查联动

导出预检查仍保持 dry-run：

- 不生成真实文件包。
- 不访问、不复制、不移动 NAS 文件。
- 不泄露真实路径。

页面和接口需要能显示：

- 可纳入导出。
- 阻塞。
- 缺失。
- 待审。
- 已驳回。
- 需转换预览。
- 暂不支持预览。

### E. 真实项目适用

验收必须覆盖：

- 105/503。
- 93/506 或当前环境另一个真实 NAS 项目。

如果真实项目数据不适合直接制造审核/整改状态，可以用隔离 smoke 项目验证写链路，但真实项目页面和只读状态必须通过。

## 4. 本批禁止

M1D 禁止：

- 新增 Hermes 能力。
- 继续 G4。
- 进入 8B / 8C / 9A。
- 开放真实 NAS 增删改查。
- 读取 PDF / Office / DWG / RVT / IFC 正文。
- 做真实 BIM 轻量化、构件级解析或正文索引。
- Agent 自动审批、自动整改、自动挂接。
- 把 catalog metadata 冒充正文 evidence。
- 返回真实 NAS 路径、raw row、SQL、token、secret、password。
- 为 105 / 503 写死逻辑。
- 修改 `docs/**`，除非主 agent 单独授权。

## 5. 交付标准

M1D 收口至少需要：

1. 真实项目文档交付和图纸交付页面能清楚表达标准状态、完整率、缺失、待审、驳回和下一步动作。
2. 缺失项补交仍使用分页远程文件选择。
3. 挂接、提交审核、审核通过、审核驳回、整改处理、关闭、重开、复审或重新补交链路可验证。
4. 关键动作后完整率和交付状态能刷新。
5. 导出预检查 dry-run 仍可用，不生成包、不触碰 NAS、不泄露路径。
6. 105/503 与至少另一个真实 NAS 项目页面可用。
7. 隔离 smoke 项目端到端写链路通过。
8. 前后端构建、健康检查、2、3、4R、6B、7A、8A、M1C 回归通过。

## 6. 主 Agent 裁决

当前 active 批次切换为：

`M1D：标准驱动交付闭环强化`

M1D 完成后，M1 平台本体稳定期可以进入阶段复盘；是否进入 M2A 需要用户再次确认。
