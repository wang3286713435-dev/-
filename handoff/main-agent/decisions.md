## 2026-05-12：二期批次一只读目录与 Agent 预览层裁决

裁决：

- 二期可以开始规划，但批次一范围固定为 `只读资产目录与 Agent 对接预览层`。
- 批次一只做前端资产目录、REST 权限证明、只读 catalog API、Agent preview / audit-ready 页面。
- 批次一不得做 Agent 直接增删改数据库、自动移动/删除/修复 NAS 文件、BIM 或文件正文向量化、自动结论、自动审批、多 Agent 调度真实业务动作。
- 批次一不得承诺客户生产级权限体系；本批只证明当前 REST 权限边界和 Agent 可见范围。
- 批次一开发前必须先由主 agent 写好开发 agent prompt 和测试 agent prompt，并明确开发 agent 使用 Ralph Loop 但只能围绕本批只读能力拆 story。
- 批次一验收必须证明接口只读、权限不泄漏、OpenAPI 可见、审计可追踪、前后端构建通过，且调用前后正式资产和 NAS 文件不被改变。
- 批次一通过后，才允许进入 selective indexing、NAS 文件正文只读抽取、Agent workflow / review suggestion；受控写操作必须最后单独设计和验收。

原因：

- 一期已经完成内部资产管理试运行底座，二期第一步应先把资产目录、权限证明和 Agent 可见范围做清楚。
- 企业 Agent 仍在开发中，过早开放写库、文件移动、正文向量化或自动审批会带来权限泄漏和数据污染风险。
- 当前用户明确要求不要拓展太多功能，先保持二期第一批小、稳、可验收。

## 2026-05-08：一期后端数据治理优先裁决

裁决：

- 一期研发策略调整为 `后端数据治理优先`，前端资产页面暂缓。
- 人工审核和运维先通过 REST API + Swagger/OpenAPI 完成。
- 正式开发采用方案 B：完整一期后端目标分三批开发、分批验收。
- 批次 1：数据底座与 NAS 扫描入库。
- 批次 2：异步任务、checksum、统计与事件流。
- 批次 3：企业 agent、审批与受控物理删除。
- NAS 扫描首批白名单为 `.rvt`、`.dwg`、`.ifc`、`.nwd`、`.nwc`、`.dxf`、`.pdf`。
- `RVT/IFC/NWD/NWC` 默认归为模型，`DWG/DXF/PDF` 默认归为图纸，`DWG` 可人工改为模型。
- 项目归属采用路径映射优先；路径/文件名精确包含项目编码可自动入库，其余进入待审核。
- 待审核队列允许人工修正项目归属、文件类型、专业和版本。
- 专业采用内置枚举 + 项目级扩展。
- checksum 异步补齐，任务采用数据库任务表 + 应用内 worker。
- agent 使用 API Key 接入，按项目范围授权，可查询、触发任务、提交申请，但不得直接修改正式数据。
- 增量同步采用审计/事件流。
- 物理删除 NAS 文件必须申请、审核、隔离 30 天，隔离期内可恢复，之后才允许永久删除。

原因：

- NAS 中历史项目命名不完全规范，必须保留人工审核和数据标注能力。
- 一期核心价值是让项目和 `.rvt/.dwg` 等 BIM/CAD 资产先被可靠治理，而不是先扩前端页面。
- agent 权限高且会参与任务触发和申请提交，必须把 API Key、项目范围授权、审计和审批边界提前固化。

## 2026-05-08：Claude Code + Ralph Loop 开发监控裁决

裁决：

- 开发 agent 固定为项目目录下的 Claude Code CLI，不再通过主 agent 创建临时子 agent 替代。
- 主 agent 负责在终端 PTY 中启动、观察和必要时中断 Claude Code 开发会话。
- 批次开发优先通过 Claude Code 的 Ralph Loop 插件运行：`/ralph-loop:ralph-loop "<prompt>" --completion-promise "<PROMISE>" --max-iterations <N>`。
- 当前项目已安装并启用 `ralph-loop@claude-plugins-official`，项目配置文件为 `.claude/settings.json`。
- 项目 `/skills` 中可见的 `ralph` skill 用于把 PRD/任务拆成 Ralph 可执行故事；开发 prompt 必须要求开发 agent 按 Ralph 的小故事和可验证验收标准推进。
- 为避免开发 agent 在 Ralph Loop setup 或后续工具调用中停在人工审批，主 agent 启动开发 agent 时使用 `claude --permission-mode bypassPermissions`，并全程监控输出。
- 若开发偏离批次 1 主线、扩前端、提前做批次 2/3、修改旧迁移或跳过权限/审计/验收，主 agent 有权中断会话，取消 Ralph Loop，优化 prompt 后重新启动。
- Ralph 完成承诺必须使用明确 `<promise>...</promise>`，且只有在功能、报告、构建、脚本、OpenAPI、审计和范围检查全部为真时才能输出。

原因：

- 用户要求开发 agent 拥有自己的 Claude Code 上下文，同时希望主 agent 实时监控进度。
- Ralph Loop 可以让 Claude Code 在同一会话中持续迭代修复，但必须有明确完成承诺和最大迭代上限。
- 普通 Claude 会话会因权限审批阻塞 `ralph-loop` setup；自动权限模式配合主 agent 监控更符合“减少人工介入”的协作要求。

## 2026-05-11：一期主线收口与 checksum 节奏裁决

裁决：

- 一期当前不再继续扩功能，进入用户验收/内部试运行准备。

## 2026-05-11：企业 Agent DB-2 对接协议冻结

裁决：

- 企业 Agent 一期 DB-2 首轮联调允许直连 MySQL 稳定 View，但只能使用 `hermes_agent_ro` 只读账号读取 `ProjectAssetView`、`FileAssetView`、`ModelAssetView`、`AuditEventView`。

- 事件流采用 Agent 拉取，主游标为 `AuditEventView.event_id`，`created_at` 只作为重叠窗口辅助字段。
- 当前 View 没有稳定权限字段，Hermes mirror 层必须默认 `DENIED`；只有平台 API Key 项目授权范围或后续明确权限字段证明可见时，资产才能进入 prompt、evidence 或 answer。
- 本机同环境联调允许读取真实项目名、文件名和 NAS 路径做字段握手，但不得持久化到长期 memory、向量库、搜索库、外部云服务、外部日志或客户材料。
- 一期 DB-2 不允许读取 PDF/Office 正文，不做 OCR、全文索引或正文向量化。
- Agent 可通过平台 API Key 在授权项目范围内触发扫描/checksum 任务、提交标注或删除申请，但不得直写数据库，不得绕过审批和审计。
- 下载接口默认不直接放开；确需调用时必须走平台授权接口、权限校验和审计，客户环境默认不返回真实 NAS 路径。
- shared-dev / staging 当前不阻塞本机联调；触发远程协作、持续测试、演示或客户交付前验证后，由平台/运维创建只读账号并通过安全渠道交付。
- 客户环境默认只做结构握手；真实样例读取必须经业务负责人和数据负责人书面确认，否则提供脱敏 View 或脱敏样本。

原因：

- 企业 Agent 能力强、权限高，如果直接把真实项目名、文件名、NAS 路径或正文内容持久化到 memory / 向量库，会造成权限泄漏和客户交付风险。
- 当前一期目标是“资产找得到、路径可追溯、权限管得住”，不是正文知识库或全量语义检索。
- 先冻结 DB-2 元数据合同，后续 DB-3 正文检索、搜索引擎、向量化和客户版安全策略可以单独设计和验收。
- 真实 NAS 项目识别先按一级目录生成项目台账和路径映射建议，二级目录作为资料类型和置信度线索。
- 目录名为 `数字-项目名` 时，项目编码优先取数字部分；数字前缀重复时必须短哈希或人工确认，不能静默冲突。
- 未编号、弱编号、投标、参考、样板、标准、临时和专题资料目录不得直接自动作为正式项目，应进入 `NEEDS_CODE_REVIEW`、`REFERENCE`、忽略或待审核清单。
- 首批真实试点目录为 `101-C塔`、`98-深圳口岸项目`、`99-丰图既有建模项目`。
- 真实 NAS 接入拆成两个 Batch 1 增强批次：`Batch 1.1: NAS 项目目录发现与真实结构适配`，`Batch 1.2: 文件类型扩展、低价值文件治理、幂等扫描`。
- Batch 1.1/1.2 仍复用 `core_projects`、`data_asset_project_path_mappings`、`data_asset_scan_tasks`、`data_asset_scan_candidates`、`data_file_resources`、`data_asset_import_jobs`、`data_asset_import_rows` 和既有 SQL View，不新增另一套导入库。
- `work-center`、`data_model_integrations`、`data_managed_objects` 只能消费已确认的正式资产，不直接读取候选记录作为正式数据源。

原因：

- 真实 NAS 已观察到 27 个一级目录，既有正式项目，也有投标、参考、临时和专题资料；如果只按扩展名粗暴扫描，容易造成项目编码冲突和资产污染。
- 多个项目复用标准二级目录，说明平台应利用目录结构做置信度和资料类型判断。
- 两个小批次能利用已完成的 Batch 1 数据底座，同时不抢 Batch 2 的异步任务、checksum、统计和事件流优先级。

## 2026-05-09：非标准 NAS 数据暂缓入库与 Agent 辅助治理裁决

裁决：

- `98-深圳口岸项目`、`95-*`、`99-*` 以及投标、参考、未知命名目录暂不导入正式资产库。
- 这些目录在当前阶段只保留在文件系统和发现报告中，不创建正式项目资产、不创建正式文件资产、不进入企业 agent 稳定读模型。
- 后续接入企业 agent 后，将这类目录作为“非标准资料治理”能力处理：先由 agent 辅助识别目录结构、文件类型、项目归属、重复编号、临时/转换资料和建议编码，再由人工确认后才允许导入数据库。
- 非标准目录治理必须全程只读 NAS；治理过程中不得移动、改名、删除 NAS 原文件。
- 治理结论必须可追溯：包括 agent 建议、人工确认人、确认时间、编码/项目拆分依据、被忽略目录原因和最终入库范围。

原因：

- `98` 可能是多个口岸样板集合，不能简单当作单一项目导入。
- `95` 与 `99` 存在重复编号，直接入库会污染项目编码、权限和后续检索。
- 投标、参考、未知命名目录不是确定的正式交付项目，不能进入正式资产库。
- 这类复杂历史资料正是平台后续数据治理能力的价值点，适合由企业 agent 辅助识别，人工监控后再入库。
