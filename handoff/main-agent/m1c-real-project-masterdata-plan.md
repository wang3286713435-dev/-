# M1C：工程主数据真实项目落地计划

更新时间：2026-05-20

## 1. 批次定位

`M1C：工程主数据真实项目落地` 是 M1B 之后的平台本体稳定批次。

M1C 的目标不是新增大模块，也不是继续 Hermes/G4，而是把当前已有的“真实项目接入向导 / 初始化向导 / 工程主数据 / 交付物标准”从模板演示能力收敛成真实 NAS 项目可用的接入流程。

一句话目标：

> 真实 NAS 项目进入平台后，用户能基于资产目录证据看懂项目缺什么、草案从哪里来、为什么只是草案，并在人工确认后生成可维护的工程主数据。

## 2. 当前问题

1. 初始化向导虽然已经改名为真实项目接入向导，但用户仍可能把模板当成真实工程结构。
2. 部位树、节点类型、交付物标准和真实 NAS 项目资产之间的证据关系不够清楚。
3. 105 项目可以作为当前样本，但后续所有真实 NAS 项目都需要同样走得通，不能为 105 写死。
4. 模板只能提供建筑机电/BIM交付基础骨架，不能冒充真实项目已完成治理。
5. 用户需要能区分：
   - 平台从 catalog metadata 里看到了什么。
   - 平台缺少什么证据。
   - 草案会创建什么。
   - 哪些内容需要项目负责人复核。

## 3. 本批必须完成

### A. 真实项目接入评估可读

接入评估必须清楚展示：

- 项目是否是真实 NAS 项目。
- 是否已有资产登记。
- 文件数量、模型/图纸/文档数量、主要扩展名、主要专业线索。
- 是否已有路径映射或扫描记录。
- 是否已有部位树、节点类型、交付物标准、目录模板。
- 当前阻塞点和下一步动作。

评估文案必须明确：

- 只读取目录级元数据。
- 不读取 PDF/Office/DWG/RVT/IFC 正文。
- 不解析 BIM 构件。
- 不触碰 NAS 文件。

### B. 草案预览必须有证据和风险

草案预览不应只是模板清单。每类草案项需要让用户知道：

- 这是模板默认骨架，还是来自目录/文件名/项目资产的线索。
- 证据等级是 catalog-only。
- 置信度或风险提示。
- 是否需要人工确认。

如果后端已有相关字段，前端必须展示；如果字段不足，可补只读响应字段，但不要大改数据库。

### C. 应用草案必须人工确认且幂等

应用草案必须满足：

- 未传 `confirmApply=true` 或等价确认字段时必须拒绝。
- 不覆盖已存在数据。
- 重复应用不会重复创建同名同编码标准项。
- 节点类型已锁定时不能自动补充会破坏锁定语义的数据。
- 写动作必须留审计。

### D. 生成后的工程主数据可维护

应用后用户必须能继续进入：

- 部位树。
- 节点类型。
- 交付物标准。
- 文档交付。
- 图纸交付。

页面应提示“草案已生成，仍需项目负责人复核”，不能表现为真实工程结构已经完全可信。

### E. 多真实项目适用

验收必须覆盖：

- 105 对应项目。
- 至少另一个真实 NAS 项目，例如 93/506 或当前环境可用的其他真实项目。

所有逻辑必须按项目资产、标准状态和权限动态判断，不允许为 105/503 写死。

## 4. 本批禁止

M1C 禁止：

- 新增 Hermes 能力。
- 继续 G4。
- 进入 8B / 8C / 9A。
- 开放真实 NAS 增删改查。
- 读取 PDF / Office / DWG / RVT / IFC 正文。
- 做真实 BIM 轻量化、构件级解析或正文索引。
- Agent 自动创建、自动审批、自动整改、自动挂接。
- 把 catalog metadata 冒充正文 evidence。
- 返回真实 NAS 路径、raw row、SQL、token、secret、password。
- 为 105 / 503 写死逻辑。
- 修改 `docs/**`，除非主 agent 单独授权。

## 5. 交付标准

M1C 收口至少需要：

1. 真实项目接入向导可读，用户能看懂“证据、缺口、草案、确认”四件事。
2. 105 与至少另一个真实项目的接入评估可打开，且不泄露真实 NAS 路径。
3. 草案预览显示证据来源、catalog-only 边界和人工确认要求。
4. 未确认时不能应用草案。
5. 已确认应用时保持幂等，不重复创建、不覆盖已有工程主数据。
6. 应用后能进入部位树、节点类型、交付物标准继续维护。
7. 文档/图纸交付、审核整改、导出预检查不回归。
8. 前后端构建、健康检查、6A、6B、7A、8A、4R 回归通过。
