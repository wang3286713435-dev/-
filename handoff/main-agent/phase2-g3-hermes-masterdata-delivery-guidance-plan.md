# G3：Hermes 平台工作型 Agent MVP

更新时间：2026-05-20

## 1. 主 Agent 裁决

G2 已收口并完成 Git checkpoint。当前不进入 `8B：BIM 引擎接入前置与轻量化任务编排骨架`。

用户最新裁决是正确的：

- Hermes 不能长期只是平台问答 AI。
- Hermes 必须作为嵌入平台的 Agent，真实帮助员工完成数字化交付工作。
- 当前重点是把 Hermes 的已有能力做强，而不是继续扩展轻量化引擎、构件解析等新模块。

因此，G3 重新定义为：

`G3：Hermes 平台工作型 Agent MVP`

G3 目标不是继续做“问答增强”，而是让 Hermes 通过平台受控能力替员工做事。

## 2. 核心边界

Hermes 可以做事，但必须通过平台做事。

允许：

- 读取平台已授权的项目、工程主数据、交付状态、资产目录和缺失项。
- 基于现有平台数据生成操作建议。
- 生成工程主数据修复草案。
- 生成交付物标准补齐建议。
- 生成文件挂接推荐方案。
- 生成整改 / 待办草案。
- 在用户明确确认后，调用平台已有 API 执行受控动作。
- 执行动作必须留下审计。

禁止：

- 直接访问数据库写表。
- 直接修改 NAS 文件。
- 绕过平台权限调用 Hermes 或外部服务。
- 未经人工确认自动挂接、自动审批、自动整改。
- 读取 PDF / Office / DWG / RVT / IFC 正文。
- 将 raw catalog row、真实 NAS 路径或文件正文写入 Hermes memory。
- 写 OpenSearch / Qdrant / MinIO documents/chunks。

## 3. G3 的 Agent 能力分层

G3 不再只停留在 `Catalog-only Q&A`。它要形成平台内的受控工具层：

### Layer A：Read Tools

只读工具，Hermes 用于理解当前项目。

至少包括：

- 获取当前项目状态。
- 获取真实项目接入评估。
- 获取工程主数据状态。
- 获取部位树、节点类型、交付物标准摘要。
- 获取文档 / 图纸交付完整率。
- 获取缺失项。
- 获取候选文件。
- 获取导出预检查摘要。

### Layer B：Plan Tools

计划工具，Hermes 替员工生成草案，但不执行。

至少包括：

- 生成工程主数据下一步计划。
- 生成部位树修复建议。
- 生成节点类型锁定建议。
- 生成交付物标准补齐建议。
- 生成缺失项补交方案。
- 生成候选文件挂接方案。
- 生成整改 / 待办草案。

### Layer C：Human-confirmed Action Tools

人工确认后执行的工具。必须走平台 API，不允许 Hermes 直接写库。

G3 可优先实现或复用：

- 应用真实项目接入向导已有 `confirmed=true` 能力。
- 应用交付治理助手已有推荐挂接能力。
- 创建整改 / 待办草案或建议项。
- 将推荐方案保存为待确认记录。

本层要求：

- 前端必须展示执行前摘要。
- 用户必须显式勾选或点击确认。
- 请求必须带 `confirmed=true` 或等价确认字段。
- 后端必须重新校验权限和项目上下文。
- 审计必须记录发起人、项目、动作、输入摘要、执行结果。

### Layer D：禁止进入的能力层

G3 不做：

- Evidence Layer 正文检索。
- Memory Layer 长期记忆写入。
- Orchestration Layer 多 Agent 自动编排。
- NAS 写操作。
- BIM 轻量化和构件级能力。

## 4. G3 必须解决的真实使用问题

员工应该能对 Hermes 说：

- `帮我看看这个项目下一步应该做什么`
- `帮我检查工程主数据还缺什么`
- `帮我生成部位树修复建议`
- `帮我看看节点类型是否可以锁定`
- `帮我检查交付物标准还缺什么`
- `帮我推荐哪些文件可以补交`
- `帮我生成一批挂接方案`
- `我确认，把这些推荐挂接执行掉`

Hermes 应该能：

- 读取当前项目上下文。
- 明确当前缺口。
- 给出可执行建议。
- 给出风险提示。
- 在人工确认后调用平台能力执行。
- 执行后告诉用户创建、跳过、失败多少条，以及下一步去哪里看。

## 5. G3 首批建议实现范围

为避免过大，G3 首批只做这 4 件事：

1. `Hermes Action Center`
   - 在 Hermes 面板中增加“可执行建议”区域。
   - 区分：只读回答、操作草案、待人工确认动作、已执行结果。

2. `工程主数据 Agent Plan`
   - Hermes 可以基于当前项目生成工程主数据补齐计划。
   - 输出：缺什么、为什么缺、建议去哪个页面、是否可生成草案。
   - 当前不自动创建部位树 / 节点类型 / 交付物标准。

3. `交付缺失项 Agent Plan`
   - Hermes 可以基于 G1 交付治理结果生成缺失项补交方案。
   - 输出：缺失项、推荐文件、推荐理由、置信度、风险。
   - 可复用已有推荐挂接接口。

4. `人工确认后执行推荐挂接`
   - 用户在 Hermes 面板或交付治理助手里明确确认后，调用已有平台批量挂接能力。
   - 不允许自动执行。
   - 执行结果必须展示并审计。

## 6. G3 不做事项

- 不做真实 BIM 轻量化。
- 不做模型转换。
- 不做构件级解析。
- 不做正文抽取。
- 不做向量库。
- 不做 NAS 文件移动 / 删除 / 重命名 / 上传。
- 不做 Agent 自动审批。
- 不做 Agent 自动整改。
- 不做 Agent 自动改工程主数据。
- 不做客户生产级自主 Agent 承诺。

## 7. 验收标准

至少满足：

1. Hermes 不再只是回答问题，页面中存在明确的“可执行建议 / 操作草案 / 人工确认执行”能力。
2. Hermes 能读取当前真实项目状态，并生成工程主数据补齐计划。
3. Hermes 能读取交付治理缺失项，并生成补交/挂接推荐方案。
4. 用户未确认时，不能执行任何写动作。
5. 用户确认后，可以调用既有平台能力执行推荐挂接。
6. 执行后展示创建、跳过、失败及失败原因。
7. 所有执行动作都有审计。
8. 105 与另一个真实 NAS 项目均可用，不允许硬编码 105。
9. 正文 / DWG / RVT / BIM 构件类问题仍返回 Missing Evidence。
10. 不泄露真实 NAS 路径、raw row、SQL、token、secret、password。
