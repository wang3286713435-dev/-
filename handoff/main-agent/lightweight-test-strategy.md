# 轻量测试策略

更新时间：2026-05-22

## 背景

当前平台已进入真实项目主线修复阶段。每个小批次都做完整浏览器逐页点击，会显著拖慢开发节奏。后续测试 agent 默认采用轻量测试策略，只确认代码可用、主线不偏、核心红线不破。

## 默认测试范围

除非主 agent 或用户明确要求，否则测试 agent 每批只做：

1. 后端构建。
2. 前端构建。
3. 健康检查。
4. 当前批次专项脚本。
5. 必要的主线回归脚本，数量控制在 1-3 条。
6. 核心 API 抽查。
7. forbidden field / 路径泄露 / 敏感字段扫描。
8. `git diff --check`。
9. 越界检查：
   - 是否修改了不该修改的目录。
   - 是否新增 Hermes / BIM / parser / indexing / NAS 写能力。
   - 是否破坏权限、审计、路径脱敏。

## 默认不做

默认不再做：

- 浏览器逐页点击。
- 多分辨率视觉巡检。
- 每个按钮、抽屉、弹窗的人工式点击回归。
- 全部历史脚本逐个跑满。
- 对非本批重点页面做长时间手工巡检。

## 什么时候必须做浏览器验收

只有以下情况才做浏览器验收：

1. 前端路由、菜单、项目工作台、UI/UX 批次。
2. 用户明确反馈页面打不开、按钮无效、卡顿、横向溢出、字段看不懂。
3. 本批修改了核心 Vue 页面交互，并且专项脚本无法覆盖。
4. 主 agent 明确要求做浏览器短验。

浏览器验收也应保持短流程：

- 只打开 1-2 个关键页面。
- 只验证核心路径是否可见、可进入、无白屏、无明显错位。
- 不做全站点击巡检。

## 回归脚本选择原则

每批最多默认跑：

- 当前批次专项脚本。
- 与本批直接相关的 1-2 条历史脚本。
- 一条安全底线脚本，如文件访问或权限隔离。

只有在提交前、合并前、重大重构后，才跑全量回归。

## 报告格式

测试报告应更短，重点写：

1. 结论：通过 / 不通过。
2. P0 / P1 / P2。
3. 执行了哪些命令。
4. 当前批次专项结果。
5. 是否发现越界。
6. 是否建议收口。

不要写大段页面描述，除非确实做了浏览器验收并发现问题。

## 对 M2D 的应用

M2D 测试重点：

- 后端构建。
- 前端构建。
- 健康检查。
- `scripts/dev/check-m2d-real-project-masterdata-onboarding.sh`。
- `scripts/dev/check-m2c-delivery-package-archive.sh`。
- `scripts/dev/check-m1c-real-project-masterdata.sh`。
- `git diff --check`。
- 接口响应 forbidden field 扫描。
- 越界检查。

M2D 默认不需要逐页浏览器点击。若要做页面检查，只打开 105 工程主数据页确认不白屏、文案不明显误导即可。
