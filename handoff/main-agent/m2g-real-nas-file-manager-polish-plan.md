# M2G：真实 NAS 文件管理器灰度完善计划

更新时间：2026-05-23

## 1. 批次定位

`M2G：真实 NAS 文件管理器灰度完善`

M2G 是 M2F 收口后的主线批次，目标是把当前平台最核心的文件管理能力从“接口可用、按钮可点”打磨成“员工愿意日常使用的文件管理器”。

本批采用产品形态：

`类 Windows / macOS 文件管理器 + 工程资料字段`

不是网盘，不是 BIM 引擎，不是 Hermes 自动治理。

## 2. 当前问题

当前文件管理器已有：

- 左目录树。
- 右文件表。
- 上传文件。
- 新建文件夹。
- 重命名文件 / 文件夹。
- 移动文件 / 文件夹。
- 删除到回收站。
- 回收站恢复。
- 操作记录。
- 受控 NAS 写入灰度。

但真实使用仍存在问题：

- 新建文件夹曾出现“请先切换到当前项目”这类项目上下文误判。
- 操作入口堆在顶部，用户不容易知道当前目录能做什么。
- 移动 / 重命名 / 删除的交互还不像文件管理器。
- 成功后目录树和文件列表刷新不够自然。
- 回收站、操作记录、失败原因还偏技术。
- 写入权限、灰度范围、当前目录是否可写需要更直观。
- 目录树、面包屑、文件表、操作按钮之间的关系需要更顺。

## 3. 本批目标

1. 项目内文件管理器始终以路由 `projectId` 为准，不再因全局当前项目不一致导致误报。
2. 新建文件夹、上传、重命名、移动、删除到回收站、恢复可以形成顺畅单项操作闭环。
3. 操作成功后，目录树、面包屑、文件列表、当前目录状态能自然刷新。
4. 当前目录是否可写、为什么不可写，要用员工能看懂的语言说明。
5. 删除统一表达为“移入回收站”，不让普通用户看到“隔离区”等技术词。
6. 失败原因脱敏、可理解，不泄露真实 NAS 路径。
7. 保持所有真实写操作都经过后端权限、灰度范围和审计。

## 4. 体验目标

员工进入文件管理页后，应能直观看懂：

- 当前在哪个项目。
- 当前在哪个文件夹。
- 当前文件夹是否可写。
- 能上传 / 新建 / 重命名 / 移动 / 删除哪些内容。
- 操作失败时为什么失败，下一步该怎么做。
- 删除后的内容在哪里恢复。

文件表默认重点字段：

- 文件名。
- 类型。
- 版本。
- 大小。
- 专业。
- 状态。
- 操作。

技术字段仍然保留在“技术信息 / 诊断信息”折叠区。

## 5. 允许范围

允许：

- 修复文件管理器项目上下文、目录刷新、操作反馈、按钮状态、弹窗文案问题。
- 优化 `AssetProjectFileBrowser.vue` 及相关组件。
- 必要时对 `ControlledNasApplicationService` 做最小 bugfix。
- 增强现有 NAS 写入 API 的响应文案和错误码，前提是不改变安全边界。
- 新增 M2G 专项脚本。

优先修复：

- 新建文件夹不应再提示“请先切换到当前项目”。
- 操作成功后应刷新目录树和文件列表。
- 移动目标最好使用目录选择，不应只靠手输路径。
- 回收站恢复后应刷新当前视图。
- 操作记录应按用户语言展示，不暴露真实路径。

## 6. 禁止范围

禁止：

- 永久删除作为普通员工能力开放。
- 绕过灰度开关、项目权限、角色权限或审计。
- 直接暴露真实 NAS 绝对路径。
- 读取 PDF / Office / DWG / RVT / IFC 正文。
- 新增 Hermes 自动操作。
- 新增 BIM 轻量化或构件解析。
- 写 OpenSearch / Qdrant / MinIO documents/chunks。
- 改动 `docs/**`。

## 7. 推荐实现重点

### 7.1 项目上下文

- 文件管理页所有 NAS 写操作必须使用路由中的项目 ID。
- 如果全局 currentProject 与路由项目不一致，前端不应直接报错；应优先按路由项目执行。
- 后端仍按 JWT 权限和项目访问校验，前端不得伪造权限。

### 7.2 文件管理器交互

- 顶部工具栏收敛为常用操作：
  - 上传文件。
  - 新建文件夹。
  - 刷新。
  - 回收站。
  - 操作记录。
- 重命名 / 移动 / 删除建议在文件行“更多”或目录节点操作中触发。
- 当前目录工具可以显示：
  - 当前目录。
  - 可写 / 不可写。
  - 不可写原因。
- 双击文件夹进入目录；面包屑可返回上级。

### 7.3 操作反馈

- 上传 / 新建 / 重命名 / 移动 / 删除 / 恢复成功后：
  - 显示清楚的成功提示。
  - 刷新当前目录文件列表。
  - 必要时刷新目录树。
  - 如果当前目录被移动或删除，应回到上级目录或项目根目录。
- 失败后：
  - 显示业务原因。
  - 不显示真实 NAS 路径。
  - 不只显示英文异常。

### 7.4 回收站

- 用户可见命名统一为“回收站”。
- 回收站列表展示：
  - 名称。
  - 类型。
  - 原目录提示。
  - 操作时间。
  - 操作人。
  - 到期时间。
  - 恢复按钮。
- 不在本批开放永久删除。

## 8. 测试策略

继续轻量测试，但 M2G 涉及文件管理器真实交互，因此需要：

- 后端构建。
- 前端构建。
- 健康检查。
- M2G 专项脚本。
- M2B NAS 写入灰度回归。
- 文件访问安全回归。
- `git diff --check`。
- 浏览器短验：只打开一个真实项目文件管理页，确认按钮、目录、回收站、操作记录不白屏、不横向溢出。

自动化脚本真实写入必须使用隔离测试项目或隔离临时目录，不得写真实业务目录。

## 9. 验收标准

M2G 通过条件：

- 新建文件夹不再出现项目上下文误判。
- 上传 / 新建 / 重命名 / 移动 / 删除到回收站 / 恢复至少在隔离目录脚本中全链路通过。
- 真实项目页面文件管理器可打开，按钮状态清晰。
- 当前目录可写状态和原因清楚。
- 操作成功后目录树和文件表刷新。
- 回收站命名统一，无“隔离区”暴露给普通用户。
- 操作记录和失败原因不泄露真实路径。
- M2B 回归不破坏。
- 未新增 Hermes / BIM / parser / indexing。

## 10. 后续建议

M2G 收口后，建议进入：

`M2H：多真实项目复制试点`

目标是将 105 的交付闭环和文件管理器体验复制到更多真实项目，验证平台不是为单一项目特化。
