# M2A：NAS 受控文件操作安全底座

## 1. 批次定位

`M2A` 是 M1F 员工注册、项目授权与局域网试运行后的下一批主线能力。

本批不是一次性开放“完整 NAS 增删改查”，而是先建立真实 NAS 写操作的安全底座，让平台能在权限、路径、审计和回滚边界清楚的前提下，逐步开放员工可用的文件操作。

当前 M2A 固定为：

`M2A：NAS 受控文件操作安全底座（第一批：低风险写操作 + 删除到隔离区，不开放永久删除与批量操作）`

## 2. 为什么不能一次性开放完整 CRUD

真实 NAS 文件是公司生产资产。一旦平台直接开放移动、删除、覆盖和批量操作，任何权限或路径校验问题都会变成真实文件丢失、误移动、误覆盖或跨项目泄露。

因此 M2A 必须先完成：

- 后端统一解析项目 NAS 根目录。
- 前端不传、不展示真实 NAS 绝对路径。
- 所有写操作限制在当前项目授权根目录内。
- 所有写操作先做权限校验、路径校验、冲突校验和审计。
- 删除只允许进入隔离区，不允许永久删除。
- 先用安全沙箱脚本验收，再进入真实项目人工试点。

## 3. 本批允许范围

允许开发：

1. 项目内新建文件夹。
2. 项目内上传文件。
3. 项目内文件 / 文件夹重命名。
4. 项目内文件 / 文件夹移动。
5. 项目内删除到隔离区。
6. 隔离区恢复。
7. 操作后同步平台元数据。
8. 操作审计和操作记录查询。
9. 文件管理页面增加受控操作入口、确认弹窗和结果反馈。

如当前没有目录元数据表，允许追加 Flyway 迁移增加平台目录记录表，用于承载平台创建的空目录，避免重新引入页面打开时同步扫描真实 NAS 的问题。

## 4. 本批禁止范围

禁止：

1. 永久物理删除。
2. 跨项目移动。
3. 批量删除、批量移动、批量重命名。
4. Agent / Hermes / 自动流程触发任何 NAS 写操作。
5. 前端传入真实 NAS 绝对路径。
6. 前端展示真实 NAS 绝对路径。
7. 绕过项目路径映射直接操作任意路径。
8. 读取 PDF / Office / DWG / RVT / IFC 正文。
9. 真实 BIM 轻量化、构件解析、正文索引。
10. 修改 `docs/**`。

## 5. 权限规则

M2A 第一批按项目角色控制：

- `VIEWER`：只读，不允许任何 NAS 写操作。
- `DELIVERY_ENGINEER`：允许新建文件夹、上传、重命名、移动。
- `PROJECT_ADMIN`：允许新建文件夹、上传、重命名、移动、删除到隔离区、恢复。

所有接口必须以服务端权限校验为准，不能只靠前端隐藏按钮。

管理员只能操作自己有权限的项目。未授权项目必须返回无权限或不可见。

## 6. 路径安全规则

所有写操作必须满足：

1. 服务端根据 `projectId` 找到项目 NAS 路径映射。
2. 前端只能传相对目录、目录 ID 或文件 ID。
3. 服务端拼接并解析 canonical path。
4. 解析后路径必须仍在当前项目根目录内。
5. 禁止 `..`、绝对路径、跨盘路径、跨项目路径。
6. 禁止通过符号链接逃逸项目根目录。
7. 默认不覆盖同名文件或目录，冲突时返回明确错误。
8. 失败原因返回前端时必须脱敏。

## 7. 元数据同步规则

- 上传文件后必须创建或更新平台文件资产记录，但不得把二进制写入数据库。
- 重命名 / 移动文件后必须保持平台文件 ID 稳定，并同步逻辑路径、文件名、更新时间等元数据。
- 新建空文件夹后必须能在目录树中显示。
- 删除到隔离区后，原文件记录应进入不可用 / 隔离状态，不能继续被交付挂接。
- 恢复后应恢复可用状态，并刷新目录和文件列表。
- checksum 可标记为待补齐或失效，不能伪造 checksum。

## 8. 前端交互要求

在项目文件管理页中增加“真实 NAS 写操作”入口，但必须有清晰风险提示：

- 新建文件夹。
- 上传文件。
- 重命名。
- 移动。
- 删除到隔离区。
- 隔离区 / 操作记录查看。

每个写操作必须：

1. 明确提示“将直接操作公司 NAS 文件”。
2. 显示项目、目标目录、文件名、操作类型。
3. 要求用户确认。
4. 显示执行结果、失败原因和 traceId / 操作编号。
5. 操作完成后刷新目录树和文件表。
6. 按角色禁用不可用按钮，并给出原因。

## 9. 推荐接口

开发 agent 应优先复用既有接口和已有隔离 / 删除审批能力。如缺少接口，可在 data-steward 范围内新增：

- `POST /api/data-steward/projects/{projectId}/nas/directories`
- `POST /api/data-steward/projects/{projectId}/nas/files:upload`
- `PATCH /api/data-steward/projects/{projectId}/nas/files/{fileId}:rename`
- `POST /api/data-steward/projects/{projectId}/nas/files/{fileId}:move`
- `POST /api/data-steward/projects/{projectId}/nas/files/{fileId}:quarantine`
- `POST /api/data-steward/projects/{projectId}/nas/quarantine/{recordId}:restore`
- `GET /api/data-steward/projects/{projectId}/nas/operations`

接口命名可按现有项目风格调整，但契约必须清晰、纳入 OpenAPI，并返回统一响应和 `traceId`。

## 10. 专项脚本要求

必须新增：

`scripts/dev/check-m2a-controlled-nas-write.sh`

脚本必须默认使用安全临时目录或测试项目映射，不得直接操作真实生产 NAS 项目目录。

脚本至少覆盖：

- 创建测试项目或测试映射。
- 创建文件夹。
- 上传小文件。
- 重命名。
- 移动。
- 删除到隔离区。
- 恢复。
- viewer 写操作被拒绝。
- 未授权项目写操作被拒绝。
- 路径穿越被拒绝。
- 响应不包含真实 NAS 路径、raw row、SQL、token、secret、password。
- 审计或操作记录可查。

## 11. 验收红线

任一情况出现即 P0：

- 操作了真实生产 NAS 项目目录但没有明确人工确认和安全边界。
- 永久删除真实文件。
- viewer 或无项目权限用户可以写 NAS。
- 可以通过 `..`、绝对路径或符号链接逃逸项目根目录。
- 可以跨项目移动 / 访问文件。
- 前端或 API 响应泄露真实 NAS 绝对路径。
- 写操作没有审计。
- Agent / Hermes 触发了 NAS 写操作。

## 12. M2A 收口条件

M2A 第一批收口必须满足：

- 后端构建通过。
- 前端构建通过。
- 健康检查通过。
- `check-m2a-controlled-nas-write.sh` 通过。
- M1C / M1D / M1E / M1F 回归通过。
- 浏览器页面在受控测试项目中可完成新建、上传、重命名、移动、隔离、恢复。
- 不泄露真实路径。
- 不发生永久删除。
- 不引入 Hermes / G4 / 8B / 8C / 9A 能力。
