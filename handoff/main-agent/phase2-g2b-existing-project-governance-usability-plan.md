# G2-B：既有真实项目治理可用性补丁

更新时间：2026-05-20

## 1. 主 Agent 裁决

`G2-A：真实项目接入与工程主数据映射修复` 已通过测试 agent 验收，当前无 P0/P1，仅剩 3 个不阻塞的 P2。

本轮已读取共享文档 `DigitalDeliveryProject/agent-briefings/hermes_capability_handoff.md`。后续 Hermes 接入按四层推进：

`Catalog Layer -> Evidence Layer -> Memory Layer -> Orchestration Layer`

G2-B 只强化 `Catalog Layer` 和常驻入口，不进入 Evidence / Memory / Orchestration 的真实能力实现。

G2-A 已解决：

- 真实 NAS 项目分类。
- 真实项目接入向导。
- 模板草案 / 待确认语义。
- 105 项目识别为真实 NAS 项目。
- Hermes project-level 问答、Missing Evidence 和路径脱敏。

但用户指出新的真实可用性问题：资产总览 Hero 区和项目入口父子结构不清晰，普通员工仍不知道“哪个功能有什么用、下一步该做什么”。这属于 G2 的治理分支收尾，不是 8B / 8C / 9A，也不是新主线。

因此启动：

`G2-B：既有真实项目治理可用性补丁`

105 项目只作为当前验收样本，不能把逻辑写死为 105。所有能力必须适用于已接管的真实 NAS 项目。

## 2. 目标

让所有既有真实 NAS 项目都能更清楚地进入数字化交付治理流程。

本批重点不是新增业务能力，而是把已有能力解释清楚、组织清楚、入口清楚：

1. 资产总览 Hero 区重排，解释“平台现在管理什么、下一步做什么”。
2. 真实项目列表父子结构更清楚，用户能区分项目状态和下一步动作。
3. 项目卡片 / 行动入口能引导进入：
   - 查看资产目录。
   - 接入评估。
   - 工程主数据草案。
   - 交付治理助手。
4. Hermes 作为全平台常驻入口出现，至少在核心工作区可随时打开。
5. Hermes 能基于当前项目和当前页面，解释页面用途、下一步动作和当前项目缺口。
6. 修正历史 smoke 项目在“全部”筛选中分类不准的问题。

## 2.1 Hermes 分层边界

G2-B 当前允许：

- Hermes 常驻入口。
- 当前页面 / 当前项目上下文注入。
- 基于资产目录的 catalog-only 问答。
- 页面用途、下一步动作、治理路径解释。
- Missing Evidence 解释。
- 低敏 feedback / memory continuity 的 UI 预留。

G2-B 当前不允许：

- `document_evidence_search`。
- PDF / Office 正文问答。
- DWG / RVT / BIM 内容理解。
- BIM 构件参数检索。
- NAS semantic collection。
- Hermes long-term memory 写入文件内容、raw catalog row 或 raw path。
- 多 Agent 编排或自动治理。

Evidence Layer、Memory Layer、Orchestration Layer 只作为后续路线，不作为 G2-B 验收项。

## 4. 资产总览 Hero 区要求

Hero 区要形成清晰父子结构：

第一层：平台目标

- 当前系统用于接管真实 NAS 项目，并推进数字化交付治理。

第二层：项目状态

- 真实 NAS 项目数量。
- 已登记资产项目。
- 已初始化主数据项目。
- 已进入交付治理项目。
- 待接入 / 待治理项目。

第三层：下一步动作

- 进入真实项目。
- 查看接入评估。
- 完善工程主数据。
- 进入交付治理助手。

第四层：风险提醒

- 缺工程主数据。
- 缺交付标准。
- 待审核。
- 缺 checksum。
- 低置信度。

要求：

- 每个入口要说明“这是干什么的”。
- 不堆技术字段。
- 不做营销 hero。
- 不新增复杂大屏。
- 不出现横向撑爆。

## 5. 真实项目治理路径

对每个真实 NAS 项目，用户应能理解以下路径：

`资产目录 -> 接入评估 -> 工程主数据草案 -> 交付治理助手 -> 缺失项解释 -> 人工确认挂接`

本批不要求自动完成上述流程，但页面上必须让用户知道：

- 当前项目在哪一步。
- 下一步应该做什么。
- 为什么要做这一步。
- 如果项目还没准备好，缺什么。

## 6. Hermes 常驻入口 MVP

Hermes 要从局部功能升级为常驻式平台助手的 MVP。

要求：

- 在平台核心工作区提供常驻入口，例如右下角浮窗或全局按钮。
- 至少覆盖：
  - 资产总览。
  - 项目工作台。
  - 真实项目接入向导。
  - 交付治理助手。
- Hermes 能接收当前页面上下文：
  - 当前路由。
  - 当前项目 ID。
  - 当前项目编码 / 名称。
  - 当前页面类型。
- Hermes 能回答：
  - 这个页面是干什么的？
  - 我下一步应该做什么？
  - 当前项目处于哪一步？
  - 为什么模板只是草案？
  - 工程主数据缺什么？
  - 交付治理助手能做什么？
  - 哪些资料还缺证据？

边界：

- 仍走平台后端 Gateway。
- 前端不直连 Hermes。
- 不读取文件正文。
- 不写 Hermes memory。
- 不自动写库。
- 不自动挂接。
- 不自动审批。
- 不触碰 NAS。

## 7. G2-B 不做事项

本批不做：

- 8B / 8C / 9A。
- 真实 BIM 轻量化。
- 真实 NAS 增删改查。
- 文件正文读取。
- 构件级解析。
- selective indexing。
- Agent 自动执行治理动作。
- 真实交付结论自动生成。
- 新的大屏或复杂驾驶舱。

## 8. 验收标准

必须满足：

1. 资产总览 Hero 区能解释平台目标、项目状态、下一步动作和风险提醒。
2. 用户能看懂“真实项目 / 待接入 / 已初始化 / 已进入治理”的父子关系。
3. 105 项目作为样本可从资产总览进入治理路径，但逻辑不硬编码 105。
4. 至少另一个真实 NAS 项目也展示同样的治理路径和入口。
5. Hermes 常驻入口在核心页面可见。
6. Hermes 能带当前页面和当前项目上下文提问。
7. Hermes 能回答页面用途和下一步动作。
8. 历史 `B6A-SMOKE-*` 项目在“全部”筛选中识别为测试项目。
9. 页面不泄露真实 NAS 路径。
10. 不触碰真实 NAS 文件。
11. 不进入 8B / 8C / 9A。

## 9. G2 收口规则

G2-B 通过后，整个 G2 必须收口，不允许继续无限扩张。

G2 收口后，再做 Git checkpoint，然后恢复二期后续主线候选：

- 8B。
- 8C。
- 9A。

是否进入哪个主线批次，由用户再次确认。
