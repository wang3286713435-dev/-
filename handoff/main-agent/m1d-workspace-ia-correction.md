# M1D 补充裁决：项目工作台信息结构纠偏

更新时间：2026-05-20

## 1. 裁决结论

本问题并入当前 `M1D：标准驱动交付闭环强化`，作为 M1D 收口前必须处理的 P1 级信息结构修复。

不新增 G 分支，不进入 Hermes，不进入 8B / 8C / 9A。

## 2. 根因

当前平台已经有项目资产、工程主数据、工作中心、Hermes、交付治理助手等能力，但它们在用户感知中仍像并列功能。普通员工无法自然理解：

1. 我先看项目里有什么文件。
2. 文件和目录线索说明项目结构可能是什么。
3. 我确认项目结构和交付标准。
4. 系统告诉我缺什么。
5. 我确认文件挂接。
6. 审核整改。
7. 交付预检查。

真正的产品顺序应该是：

`先定义规则 -> 再按规则交付`

## 3. 目标信息结构

项目工作台应呈现为：

```text
项目工作台
├─ 项目资产
│  ├─ 文件目录
│  ├─ 文件列表
│  └─ 文件详情
│
├─ 工程主数据
│  ├─ Agent 接入评估
│  ├─ 工程主数据草案
│  ├─ 部位树
│  ├─ 节点类型
│  └─ 交付物标准
│
└─ 交付工作中心
   ├─ 交付治理助手
   ├─ 文档交付
   ├─ 图纸交付
   ├─ 审核整改
   └─ 交付包预检查
```

说明：

- `工作中心` 不是 `工程主数据` 的子功能。
- `工作中心` 必须属于当前项目工作台。
- `工作中心` 在信息顺序上必须排在 `工程主数据` 之后。
- Hermes 不能再作为主线入口放大；交付治理助手可以保留为工作中心内的辅助入口，但不得成为主线依赖。

## 4. 准入规则

### A. 没有工程主数据

工作中心仍可见，但页面和入口必须提示：

> 请先生成 / 确认工程主数据草案。

需要引导用户回到：

- 真实项目接入评估。
- 工程主数据草案。
- 部位树。
- 节点类型。
- 交付物标准。

文档交付、图纸交付、审核整改、交付包预检查不能表现为“已经可正常交付”。

### B. 工程主数据已确认

工作中心开放：

- 文档交付。
- 图纸交付。
- 审核整改。
- 交付包预检查。

页面继续按 M1D 原目标展示：

- 应交项。
- 缺失项。
- 补交。
- 审核。
- 整改。
- 复审或重新补交。
- 完整率刷新。
- dry-run 导出预检查。

## 5. 并入 M1D 的开发要求

开发 agent 需要在 M1D 当前改动基础上补齐：

1. 项目工作台顶部或模块区按 `项目资产 -> 工程主数据 -> 交付工作中心` 展示。
2. `交付工作中心` 不作为工程主数据子页面，但必须在视觉顺序和文案上排在工程主数据之后。
3. 工作中心入口根据工程主数据就绪状态展示准入提示。
4. 工程主数据未就绪时，文档交付 / 图纸交付页面应保留安全可读页面，但明确阻塞原因和下一步入口。
5. 工程主数据已就绪时，文档交付 / 图纸交付页面继续执行 M1D 标准交付闭环。
6. 不改变项目路由大结构，不大改后端模型。
7. 不把 Hermes 变成必经入口。

## 6. 验收补充

测试 agent 在 M1D 验收中必须补充：

1. 项目工作台能看出三段顺序：`项目资产 -> 工程主数据 -> 交付工作中心`。
2. 工作中心不在工程主数据下面，但排在工程主数据之后。
3. 对缺少工程主数据的项目，工作中心提示先生成 / 确认工程主数据草案。
4. 对工程主数据已就绪的项目，文档交付、图纸交付、审核整改、交付包预检查可进入。
5. 页面主路径不依赖 Hermes。
6. 105/503 与 93/506 至少两个真实项目都适用。

## 7. 收口口径

M1D 不能只收口后端交付状态字段或页面卡片；必须同时收口这次信息结构纠偏。

否则用户仍会觉得平台“功能很多但不知道顺序”，M1D 不应判定完成。
