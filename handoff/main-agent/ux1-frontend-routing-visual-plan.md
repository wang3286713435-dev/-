# UX1：前端路由逻辑与视觉体验专项优化批次

## 1. 批次定位

`UX1` 是 M2B 之后插入的前端专项优化批次。

固定顺序：

`M2A 当前 bugfix 收口并推送 Git -> M2B 受控 NAS 写操作真实项目灰度 -> UX1 前端路由与视觉体验优化`

UX1 不抢占 `M2B` 命名，不提前启动真实 BIM 轻量化，不继续 Hermes/G4，不改变后端能力边界。

## 2. 开发方式

- UX1 由 `Gemini` 主导开发。
- 主 agent 负责监控、审计、纠偏和最终收口判断。
- 测试 agent 负责专项验收和主线回归。
- Gemini 完成后必须写入 `handoff/dev-agent/latest-report.md`。

## 3. 固定范围

UX1 只允许调整前端：

- 路由。
- 菜单。
- 项目工作台。
- 组件布局。
- 样式。
- 交互文案。
- 页面体验。

UX1 禁止：

- 修改 `backend/**`。
- 修改数据库迁移。
- 新增或修改后端接口。
- 改变权限规则。
- 引入 Hermes 新能力。
- 引入 BIM 轻量化。
- 新增真实 NAS 写能力。
- 删除旧链接导致历史书签或脚本失效。

## 4. 产品目标

当前平台核心功能已经基本具备，但用户体验问题明显：

- 跳转逻辑复杂。
- 全局当前项目与路由项目容易混淆。
- 项目资产、工程主数据、工作中心的前后关系不够清楚。
- 页面视觉不精致，企业中后台质感不足。
- 员工不知道每个页面做什么、下一步应该点哪里。

UX1 要把平台从“功能能用”提升到“普通员工能理解、愿意用、少迷路”。

## 5. 路由与导航要求

- 主入口仍为 `/data-steward/assets`。
- 用户先进入资产总览，再进入具体项目工作台。
- 项目工作台内所有能力围绕路由 `projectId` 展开。
- 保留旧全局链接兼容跳转。
- 旧 `/master-data/*`、`/work/*`、`/data-steward/models` 等入口应自动跳转到当前项目工作台内对应页面。
- 不允许直接删除旧路由。
- 刷新项目内任意页面后，项目工作台导航必须保持正确。

## 6. 信息结构要求

项目工作台必须清晰呈现：

`项目资产 -> 工程主数据 -> 交付工作中心`

每个区域要让员工知道：

- 这个模块是做什么的。
- 当前项目处于什么状态。
- 下一步应该去哪里。
- 哪些能力尚不可用或需要先完成前置条件。

工作中心不是工程主数据的子功能，但必须在项目工作台下排在工程主数据之后。

## 7. 视觉方向

视觉方向固定为：

`精致企业中后台`

要求：

- 保留 Vue 3 + Element Plus 体系。
- 不做炫酷大屏化。
- 不做营销式 landing page。
- 不做大面积装饰渐变和无意义卡片堆叠。
- 优化菜单、顶部项目导航、标题区、状态卡、操作区、表格、抽屉、弹窗、空状态和错误提示。
- 页面应更清爽、更有层级、更适合员工日常高频使用。

## 8. 重点页面

UX1 优先精修：

- 资产总览。
- 项目工作台。
- 文件管理。
- 工程主数据初始化 / 接入评估。
- 部位树。
- 节点类型。
- 交付物标准。
- 文档交付。
- 图纸交付。
- 审核整改。
- 交付包预检查。

## 9. 验收红线

任一情况出现即 P0：

- 修改后端业务逻辑。
- 删除旧链接导致历史路由不可用。
- 项目内页面刷新后丢失项目上下文。
- 真实 NAS 路径泄露。
- 进入 Hermes/G4/8B/8C/9A 新能力。
- 破坏 M2B/M2A/M1F/M1E/M1D/M1C 回归。

## 10. 收口条件

UX1 收口必须满足：

- 前端构建通过。
- Fresh login 后进入资产总览。
- 旧全局链接兼容跳转可用。
- 105/503 与 93/506 项目工作台可用。
- 项目资产、工程主数据、交付工作中心入口清晰。
- 页面无横向溢出、白屏、卡死和主要按钮不可见。
- 视觉和文案明显优于当前版本。
- 确认无后端文件改动。
