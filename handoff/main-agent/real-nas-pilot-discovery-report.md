1. 项目编码不能直接作为唯一键。
   `95` 和 `99` 都已重复，真实导入时必须允许人工确认唯一项目编码。

   该目录文件量大，且路径中存在大量临时/转换痕迹，应先进入待审核或低置信策略。

4. `98-深圳口岸项目` 可能需要拆成多个子项目。
   目录下包含多个口岸样板文件夹，正式入库前需要确认是一个总项目还是多个项目资产。

5. 当前挂载不是强制只读。
   本轮没有写 NAS，但全量导入前建议改用只读账号或只读挂载。

## 8. 本轮结论

真实 NAS 结构可以接入平台，但不能简单按一级目录全量自动入库。

一期平台现有的“高置信自动入库 + 低置信待审核 + 编号冲突人工确认 + SQL View/agent 读模型”路线是必要的，尤其适合当前 NAS 中存在的重复编号、临时目录、投标资料、样板目录和多子项目集合问题。

平台 `dry-run` 已验证通过，建议下一步用 `105`、`100`、`101` 三个项目做第一批数据库元数据写入；`99` 仅做压力 dry-run 和低置信审核策略验证，暂不进入第一批正式写库。
