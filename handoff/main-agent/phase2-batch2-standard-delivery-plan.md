# 二期批次二规划：标准驱动交付闭环最小可用版

更新时间：2026-05-14

## 1. 结论

二期批次二建议正式命名为：

`标准驱动交付闭环最小可用版`

本批目标不是做模型轻量化、构件级能力或企业 Agent 写操作，而是把平台从“资产可查”推进到“交付是否完整可判断”。

本批核心验收链路固定为：

`项目上下文 -> 部位树 -> 节点类型锁定 -> 交付物标准 -> 交付物类型 -> 文件资源 -> 文档/图纸挂接 -> 交付完整率 -> 缺失项清单 -> 审计留痕`

## 2. 与前置批次的关系

已完成能力：

1. 一期内部 BIM/CAD 资产库、NAS 元数据接管、数据质量与只读 Agent 合同。
2. 二期批次一只读资产目录、REST 权限证明、Agent preview / audit-ready 页面。
3. 主数据已有部位树、节点类型、交付物定义、交付物类型、交付物属性、目录模板。
4. 工作中心已有文档/图纸交付绑定和基础交付视图。

本批只补齐当前缺口：

1. 标准状态对下游交付页的前置约束更清楚。
2. 文档/图纸交付页不只显示“已挂接文件”，还要显示“按标准应该交什么、缺什么、完成多少”。
3. 交付完整率和缺失项需要后端稳定接口，而不是前端临时猜测。
4. 新增交付绑定后要能影响完整率，并写审计。

## 3. 本批范围

### 3.1 后端

建议新增或增强以下能力：

1. 交付完整率接口：
   - `GET /api/work-center/projects/{projectId}/delivery-completeness`
   - 参数：`viewType=DOCUMENT|DRAWING`、`targetType=SECTION|OBJECT`、`onlyMissing=true|false`
   - 返回：总要求数、已完成数、缺失数、完成率、缺失项列表、已完成项列表。
2. 缺失项计算规则：
   - `viewType=DOCUMENT` 对应文件类型 `DOCUMENT`。
   - `viewType=DRAWING` 对应文件类型 `DRAWING`。
   - 交付要求来自当前项目启用且必交的交付物定义，以及其下启用的交付物类型。
   - `bindingStrategy=SECTION_NODE` 按启用部位生成要求。
   - `bindingStrategy=MANAGED_OBJECT` 或 `OBJECT` 按启用管理对象生成要求。
   - 当前已有绑定且 `binding_status='BOUND'`、未删除、文件未删除、文件类型匹配，则视为完成。
3. 标准前置检查：
   - 若部位树、锁定节点类型、交付物定义、交付物类型缺失，完整率接口要返回清晰状态和缺口提示。
   - 不允许返回 500 或空洞数据让前端猜。
4. 审计：
   - 创建/删除交付绑定已有审计的，保持。
   - 如新增完整率导出或批量动作，本批不做。
5. OpenAPI：
   - 新增接口必须进入 Swagger/OpenAPI。

### 3.2 前端

建议增强现有页面，不新开大模块：

1. `/work/document-delivery`
2. `/work/drawing-delivery`

页面应展示：

1. 标准状态提示：部位、节点类型、交付物标准、目录模板是否就绪。
2. 交付完整率卡片：总要求、已完成、缺失、完成率。
3. 缺失项列表：目标部位/对象、交付定义、交付类型、文件类型、缺失原因。
4. 已完成/已挂接列表：保留现有交付视图。
5. 从缺失项快速打开“新增挂接”并预填交付物类型和目标。

性能约束：

1. 不允许在大项目里一次性拉全量文件资源。
2. 挂接文件选择应使用分页或关键词检索。
3. 加载中页面仍能切换菜单，不得卡死。

### 3.3 脚本

新增专项验收脚本：

`scripts/dev/check-phase2-batch2-standard-delivery.sh`

脚本至少覆盖：

1. 标准未就绪时接口返回可解释缺口。
2. 准备部位、节点类型、交付定义、交付类型、文件资源后，完整率先显示缺失。
3. 创建文档交付绑定后，文档完整率增加。
4. 创建图纸交付绑定后，图纸完整率增加。
5. 删除绑定后，完整率回落。
6. 普通用户无权限项目不能查询完整率或创建绑定。
7. 关键写动作可查审计。

## 4. 明确禁止事项

本批不做：

1. 模型轻量化在线预览。
2. 构件级解析、构件级搜索、碰撞检查。
3. PDF/Office/CAD/BIM 正文抽取。
4. 向量库、搜索引擎写入。
5. 企业 Agent 自动审批、自动修复、自动删除、自动写库。
6. 审核流完整实现；本批只保留现有 `reviewStatus` 展示，不新增审批工作流。
7. 客户生产级权限承诺。
8. 真实 NAS 文件移动、删除、改名或正文读取。

## 5. 开发 Agent 交付要求

开发 agent 必须使用 Ralph Loop 推进，并把 story 拆小：

1. 后端完整率计算接口。
2. 标准前置状态与错误码。
3. 前端文档/图纸交付页完整率与缺失项。
4. 文件选择分页/检索，避免大项目卡顿。
5. 专项验收脚本。
6. 报告写入 `handoff/dev-agent/latest-report.md`。

完成承诺：

`<promise>PHASE2_BATCH2_STANDARD_DELIVERY_COMPLETE</promise>`

只有当后端构建、前端构建、健康检查、专项脚本、页面冒烟、权限检查、OpenAPI 检查和报告全部完成后，才允许标记完成。

## 6. 测试 Agent 验收要求

测试 agent 必须重点验证：

1. 标准不完整时，页面和接口都能解释缺什么。
2. 绑定前缺失项存在，绑定后完成率提升。
3. 文档和图纸两条链路独立成立。
4. 权限隔离不回归。
5. 大项目页面不全量加载文件、不明显卡顿。
6. 本批没有越界进入模型轻量化、正文抽取、Agent 写操作或审核流。
7. OpenAPI 包含新增接口。

P0：

1. 无权限用户可查询或修改其他项目交付数据。
2. 完整率计算错误到无法验收。
3. 创建绑定绕过项目、文件类型或目标校验。
4. 本批接口或页面读取/修改真实 NAS 原文件。
5. 主页面白屏或无法打开。

P1：

1. 缺失项列表与绑定结果不一致。
2. 标准前置缺口提示混乱。
3. 大项目文件选择仍一次性加载全量文件。
4. 文档/图纸链路互相串数据。

## 7. 收口标准

本批通过后，平台应具备一个可向用户解释的最小交付闭环：

`标准已经定义 -> 系统知道应该交什么 -> 用户挂接文件 -> 系统知道还缺什么 -> 完整率可追踪 -> 操作可审计`

批次二通过后，才建议进入批次三：

`审核流、整改闭环、报表导出`
