# Local PR: 真实 NAS 项目文件导入后端数据库建议

## 给主 Agent 的结论

这份 PR 不是要求立刻改主线交付逻辑，而是把公司真实 NAS 项目结构转成后续开发 agent 可执行的导入口径。建议主 agent 将它作为 `一期后端数据治理` 的补充裁决，纳入下一轮 `dev-agent/current-prompt.md` 或 `test-agent/current-prompt.md`。

当前真实 NAS 路径为：

`/Volumes/zyzn/卓羽智能项目`

用户原始描述里的 `卓羽智能项目文件夹` 在共享盘中实际对应的是 `卓羽智能项目`。该目录目前可见 27 个一级项目目录，既包含正式项目，也包含投标、参考、临时或专题资料目录。

## 真实项目结构观察

一级目录大致分成四类：

- 编号项目：如 `101-C塔`、`98-深圳口岸项目`、`99-丰图既有建模项目`、`115-深圳宝安国际机场T2...`。
- 投标/参考资料：如 `投标资料`、`投标项目文件夹（实施方案参考）`、`清华斯维尔围标项目`。
- 未编号或弱编号项目：如 `深城交`、`中安投标项目`、`2024.9.3-大水灌`。
- 活跃专题目录：如 `114-香港项目`、`116-港中文（深圳）医学院智能化`。

多个真实项目已复用标准目录结构：

- `00_工作进度`
- `01_文件收发` 或 `01-文件收发`
- `02_项目资源`
- `03_过程文件`
- `04_共享文件`
- `05_发布文件`
- `06_归档文件`
- `07_浏览动画`

这说明真实数据适合按“一级目录识别项目、二级目录识别资料类型”的方式接入，而不是只按文件扩展名粗暴扫描。

## 建议导入策略

主 agent 后续派发开发任务时，建议采用“影子导入”：

1. 只读扫描 NAS，不移动、不改名、不删除原文件。
2. 先生成项目台账和路径映射，写入 `core_projects` 与 `data_asset_project_path_mappings`。
3. 扫描文件时先写 `data_asset_scan_candidates`。
4. 高置信文件再写 `data_file_resources`。
5. 低置信文件保留 `PENDING`，由人工审核后再入正式资产。
6. `work-center`、`data_model_integrations`、`data_managed_objects` 只消费已确认的正式资产，不直接读取候选记录。

这条路线和当前已完成的 Batch 1 设计一致，不会冲击主线。

## 项目编码规则

建议主 agent 明确以下编码规则，避免开发 agent 自行猜：

- 目录名前缀是 `数字-项目名` 时，项目编码优先使用数字部分，例如 `101-C塔` -> `101`。
- 同一个数字前缀重复时，不要只用数字作为唯一编码，应生成稳定编码，例如 `95-鹏投泰润学府项目` 与 `95-大鹏新区...` 可转为 `95-<短哈希>` 或人工确认编码。
- 没有数字前缀的目录不要自动作为正式项目编码，应进入 `NEEDS_CODE_REVIEW` 清单。
- 投标、参考、样板、标准类目录不要直接挂到单项目，应标记为 `REFERENCE` 或由主 agent 决定是否进入企业知识库。

## 文件分类规则

第一阶段扫描范围保持当前 Batch 1 白名单：

- `.rvt`、`.dwg`、`.ifc`、`.nwd`、`.nwc`、`.dxf`、`.pdf`

第二阶段可扩展真实项目常见文件：

- `.doc`、`.docx` -> `DOCUMENT`
- `.xls`、`.xlsx` -> `SPREADSHEET`
- `.ppt`、`.pptx` -> `PRESENTATION`
- `.glb`、`.gltf` -> `MODEL_VIEWER`
- `.zip`、`.rar` -> `ARCHIVE`
- `.wps` -> `DOCUMENT`

分类建议：

- `.rvt`、`.ifc`、`.nwd`、`.nwc` 默认 `MODEL`
- `.dwg`、`.dxf`、`.pdf` 默认 `DRAWING`
- `周报`、`汇报`、`会议纪要`、`合同`、`招标文件` 应优先进入 `DOCUMENT` 或 `PRESENTATION`
- `成果文件`、`发布文件`、`归档文件` 目录中的内容可提高置信度
- `新建文件夹`、`临时文件`、`转换` 目录中的内容默认进入待审核，不自动进入正式交付资产

## 低价值与风险文件

以下文件必须只进入候选或忽略清单，不得自动入正式资产：

- `.DS_Store`
- `Thumbs.db`
- `desktop.ini`
- `~$` 开头的 Office 锁文件
- 空文件
- 明显临时目录下的重复中间文件

建议候选记录的 `review_message` 写明原因，例如：

- `系统缓存文件`
- `Office 锁文件`
- `临时目录文件`
- `项目归属不明确`
- `目录编号冲突，需人工确认`

## 三个试点项目

建议先用以下三个项目做真实 NAS 试点：

- `101-C塔`：业务跨度最大，覆盖 BIM、智能化、驾驶舱、汇报 PPT、调研材料。
- `98-深圳口岸项目`：目录结构最标准，覆盖多口岸样板、周报、摄像头表、三维展示文件。
- `99-丰图既有建模项目`：版本、转换、整改、IFC/RVT 文件密集，适合验证重复文件和临时目录治理。

这三个项目能覆盖主线最关心的三类场景：标准化目录、复杂项目资料、模型版本治理。

## 数据库落点

建议落点保持如下，不新增另一套导入库：

- 项目台账：`core_projects`
- 项目 NAS 路径：`data_asset_project_path_mappings`
- 扫描任务：`data_asset_scan_tasks`
- 扫描候选：`data_asset_scan_candidates`
- 正式文件资产：`data_file_resources`
- 导入批次：`data_asset_import_jobs`
- 导入行结果：`data_asset_import_rows`
- 稳定查询视图：`ProjectAssetView`、`FileAssetView`、`ModelAssetView`、`AuditEventView`

不要让企业 agent 或工作中心直接读 `data_asset_scan_candidates` 作为正式数据源。

## 建议给开发 Agent 的任务边界

建议下一轮 dev-agent prompt 只做以下后端增强：

- 增加 NAS 一级目录发现接口，支持 dry-run，输出项目编码建议、路径映射建议、需人工确认目录。
- 增强扫描分类，从当前模型/图纸扩展到文档、表格、汇报、归档包和展示模型。
- 增强低价值文件识别，将缓存、锁文件、临时目录文件从自动入库降级为候选或忽略。
- 扫描入库做幂等：同项目同 `storage_uri` 重扫只更新元数据和 `last_verified_at`，不重复插入。
- 审计记录增加真实 NAS 导入动作：目录发现、映射生成、自动入库、人工审核、拒绝。

明确不做：

- 不解析模型构件。
- 不搬迁文件到 MinIO。
- 不生成轻量化模型。
- 不改 `work-center` 主线交付绑定逻辑。
- 不对 NAS 做任何写入、删除、移动或改名。

## 建议验收

验收脚本应覆盖：

- 用临时目录模拟 `101-C塔`、`98-深圳口岸项目`、`99-丰图既有建模项目`。
- dry-run 能识别三个项目和路径映射建议。
- 执行导入后能在 `core_projects` 和 `data_asset_project_path_mappings` 查到项目与路径。
- 扫描后高置信文件进入 `data_file_resources`。
- `.DS_Store`、`Thumbs.db`、`~$xxx.docx` 不自动进入正式资产。
- `FileAssetView` 能按项目、文件类型、来源 `NAS_SCAN` 查询结果。
- 现有 `check-bim-asset-batch1.sh` 不回归。

## 给主 Agent 的建议裁决

请主 agent 将真实 NAS 接入拆成两个小批次：

1. `Batch 1.1: NAS 项目目录发现与真实结构适配`
2. `Batch 1.2: 文件类型扩展、低价值文件治理、幂等扫描`

这样能利用现有 Batch 1 数据底座，不抢 Phase B 的异步任务、checksum、统计与事件流优先级。
