# M2F：真实项目交付闭环试运行计划

更新时间：2026-05-22

## 1. 批次定位

`M2F：真实项目交付闭环试运行`

M2F 是 M2E 收口后的主线批次。目标不是继续增加新模块，而是把 `105 / 503` 真实 NAS 项目中已经人工确认的工程主数据真正放进交付链路里跑通。

核心链路：

`真实资产 -> 工程主数据 -> 交付标准 -> 文档/图纸应交项 -> 文件挂接 -> 审核/整改 -> 交付包草案`

## 2. 启动背景

当前已经完成：

- 真实 NAS 项目资产接入。
- 105 演示模板清理。
- 真实资产驱动的工程主数据草案。
- 人工确认后生成正式部位树、节点类型、交付物定义、交付物类型和目录线索。
- 交付包 / 档案目录草案能力。

但还需要确认：

- 工程主数据是否真的能驱动文档 / 图纸交付页面。
- 缺失项、挂接、审核、整改和交付包草案是否形成一条员工能理解的闭环。
- 页面是否仍存在“看起来有规则、实际不知道怎么用”的断点。

## 3. 本批目标

1. 验证 105 项目正式工程主数据能驱动交付页面。
2. 文档交付和图纸交付能展示基于正式规则的应交项、缺失项、阻塞原因。
3. 缺失项解释必须清楚说明：
   - 属于哪个部位 / 交付对象。
   - 对应哪个交付定义。
   - 需要什么类型文件。
   - 为什么当前算缺失或不可交付。
4. 文件补交 / 挂接流程保持人工确认，不允许自动挂接。
5. 审核、驳回、整改、复审链路可用。
6. 交付包 / 档案目录草案能反映当前交付状态。
7. 页面不能再把模板、草案或 catalog metadata 误表达成真实正文证据。

## 4. 允许范围

允许做：

- 修复 M2E 后交付链路中的实际可用性问题。
- 调整前端文案、提示、入口和空状态。
- 补齐后端只读聚合或轻量查询，前提是不改变既有业务边界。
- 增强现有脚本，新增 M2F 专项脚本。
- 修复文档 / 图纸交付、整改、交付包草案页面中与真实项目链路直接相关的问题。

优先复用：

- 现有文档 / 图纸交付接口。
- 现有批量挂接能力。
- 现有审核 / 整改能力。
- 现有交付包草案与预检查能力。

## 5. 禁止范围

本批禁止：

- 真实 NAS 文件写入、移动、删除、重命名、复制。
- 读取 PDF / Office / DWG / RVT / IFC 正文。
- 新增 Hermes 能力。
- 接入 BIM 引擎或构件解析。
- 新增 parser / writer / indexing / 向量库。
- Agent 自动挂接、自动审核、自动整改。
- 让模板数据冒充真实工程结构。
- 启动 8B / 8C / 9A。

## 6. 开发建议

开发 agent 应先做链路审计，再做最小修复：

1. 查询 105 的标准状态、部位树、节点类型、交付物定义、交付物类型。
2. 查询文档 / 图纸交付视图，确认是否正确基于正式规则生成应交项。
3. 查询缺失项和挂接候选，确认解释是否足够业务化。
4. 查询审核 / 整改 / 交付包草案接口，确认能反映当前正式规则。
5. 发现断点时，只修当前断点，不扩展新能力。
6. 新增 `scripts/dev/check-m2f-real-project-delivery-loop.sh`，覆盖核心闭环。

## 7. 验收标准

M2F 通过条件：

- 105 工程主数据已确认状态被交付页面正确识别。
- 文档 / 图纸交付能展示应交项或合理阻塞说明。
- 缺失项解释清晰，不再只有技术字段。
- 能人工选择文件补交 / 挂接，且不能绕过确认。
- 审核 / 驳回 / 整改 / 复审链路不回归。
- 交付包草案 / 档案目录能反映交付当前状态。
- 未触碰真实 NAS 文件。
- 未读取正文。
- 未泄露真实 NAS 路径或敏感字段。
- M2E / M2D / M2C 相关回归通过。

## 8. 测试策略

继续采用轻量测试：

- 后端构建。
- 前端构建。
- 健康检查。
- M2F 专项脚本。
- M2E、M2C、文件访问安全相关回归。
- `git diff --check`。
- 仅在脚本无法覆盖页面可用性时，做 105 项目交付页面短浏览器验收。

## 9. 收口标准

M2F 收口后，平台应达到：

`105 真实项目不再只是“资产和规则都存在”，而是能按正式规则进入文档 / 图纸交付、审核整改和交付包草案链路。`

M2F 收口后，建议进入：

`M2G：真实 NAS 文件管理器灰度完善`
